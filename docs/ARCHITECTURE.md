# Architecture

## Overview

Kernova is a macOS application for creating and managing virtual machines via Apple's Virtualization.framework, supporting both macOS and Linux guests. It is built as a pure-AppKit app (the app target no longer imports SwiftUI), targeting macOS 26 (Tahoe) with Swift 6 strict concurrency. There are no third-party (non-Apple) package dependencies — Apple system frameworks plus Apple's own `swift-protobuf` package, consumed by the local `KernovaKit` package (see Dependencies).

> The clipboard subsystem (host↔guest copy/paste) is governed by a dedicated set of design principles in [CLIPBOARD.md](CLIPBOARD.md) — the authoritative *why* and rules behind the clipboard files mapped below. Consult it before changing any clipboard code.

## Component Map

### App Layer

**Files:** `AppDelegate.swift`, `HostAgentStatusItemController.swift`, `MainWindowController.swift`, `SettingsWindowController.swift`, `DetailContainerViewController.swift`, `VMDisplayWindowController.swift`, `VMToolbarManager.swift`, `GuestAgentDiskMenuItem.swift`, `ClipboardWindowController.swift`

**Launch model — resident menu-bar app (#460).** Kernova is an ordinary resident app (a single `.app` / single binary), NOT launchd-managed — the launchd LaunchAgent / `--background` / launcher-agent split was dropped in #460. `AppDelegate.main()` reads only `isTestHost` (the XCTest environment): the test host is a plain `.regular` foreground app that idle-terminates (legacy behavior, so the existing suite is unaffected); otherwise it sets `.accessory` and installs the resident `AppDelegate`. The resident app launches headless (`.accessory`), loads the VM library **without** auto-starting any VM, and installs an always-on menu-bar status item (`HostAgentStatusItemController`) — the only visible "it's running" affordance while headless, with an "Open Kernova" summon, the live running-VM list, and Quit. Whether a window appears at launch is decided by **cold-launch resolution** (`resolveColdLaunch(from:)`): the launch `oapp` Apple event's `keyAEPropData` property carries `keyAELaunchedAsLogInItem` on a login-item launch, so most launches classify deterministically — a plain `oapp` (manual launch) shows the library window immediately; the login-item property stays headless. The property is legacy-documented (Cocoa Scripting Guide "may contain") and field-proven against `SMAppService.mainApp` (Electron ships the same check); there is no modern purpose-built launch-provenance API (Apple FB10207829), so an unreadable event falls back to the **activation-settle heuristic**: a manual launch (which activates the app) shows the window on `applicationDidBecomeActive`, while a launch that stays unactivated for a ~1.5 s settle window latches headless. The decisions are the pure, unit-tested `launchProvenance(eventID:launchPropData:)` and `coldLaunchOutcome(showWindow:alreadyResolved:)`, resolved exactly once. It summons the GUI on demand — `applicationShouldHandleReopen` (a Finder reopen / Dock click / `open` of the already-running instance; macOS routes it to the single running instance rather than spawning a second, so no instance lock is needed) or the status-item commands — by morphing `.accessory`→`.regular`, then `activate(ignoringOtherApps:)` + show the summoned surface on the next runloop tick (the FB7743313 menu-bar quirk). That surface is the library window for every path except the status item's per-VM command, which opens **only** what that VM asks for: its own pop-out/fullscreen display window when that's its persisted `displayPreference` and it can present one, else the library selected on it — the pure, unit-tested `statusItemOpenTarget(displayPreference:canUseExternalDisplay:)` decides, and neither direction restores the library, other VMs' display windows, or any other window alongside it. The Dock icon then tracks window presence: every window-open forces `.regular` (`ensureRegularActivationIfAgent`), and a single global `NSWindow.willCloseNotification` observer (installed once in `startResidentApp`) schedules a next-runloop-tick reconcile (`scheduleAgentActivationPolicySync` → `syncAgentActivationPolicy`, keyed on `hasVisibleUserWindow`; deferred so the closing window has left the window list) on every window close — tracked (main library, display, clipboard, settings) or not — dropping to a headless `.accessory` app only once none remain visible, with any running VMs executing headless. `hasVisibleUserWindow` counts the app's own tracked windows plus any other visible, normal-level, titled `NSApp.windows` entry (`AppDelegate.isUntrackedUserPanel`) — the standard About panel is the motivating untracked AppKit-owned panel — so one being the last window on screen also keeps the Dock icon (#437; excludes the always-present, non-`.normal`-level `NSStatusBarWindow`; the toolbar customization palette doesn't need this path since AppKit presents it as a sheet attached to its target window, which is already one of the tracked windows). Hide (⌘H) is deliberately not special-cased — it closes no window, so no reconcile fires and the Dock icon persists. The resident app never idle-quits (`terminateIfIdle` runs only in the test host). **GUI-origin quits do not kill the app** (while *Keep Running in Menu Bar* is on — the default, #624): ⌘Q / the app menu's "Close All Windows" / the Dock's Quit are all downgraded — a gate at the top of `applicationShouldTerminate` (`!isTestHost && !quitShouldTerminateAgent`) returns `.terminateCancel` and instead (deferred) calls `closeAllGUIWindows()`, closing every user-facing window, then settling the Dock-presence reconcile synchronously (`syncAgentActivationPolicy`) so the app has already dropped to headless `.accessory` **before** it shows the transient status-item reminder popover that it's still resident (`HostAgentStatusItemController.showSoftQuitReminder`, `MenuBarQuitReminderViewController`; suppressible via "Stop Reminding Me" → `AppPreferences.menuBarQuitReminderDismissed`) — ordering that matters because the `.regular`→`.accessory` flip re-hosts the menu-bar status item, and a popover anchored before the flip is torn down with the old status window (it flashed for a frame and vanished). It takes the deliberate-exit path — the save-suspend `applicationShouldTerminate` below — only for an **authorized** termination: the app menu's honest "Quit Kernova" or the menu-bar status item's Quit (both via `requestFullQuit`, latching `userRequestedAgentQuit`), a system logout/restart/shutdown **or a scripted/programmatic quit** (`externalQuitRequiresTermination`, detected synchronously by the quit event's sender), or a TCC revocation (`terminationIsTCCRevocation`) — or whenever the preference is off (`!keepInMenuBarOnQuit`), which makes ⌘Q mean quit. **The app menu's quit section is honest per mode** (the pure `appMenuQuitItems(isTestHost:keepInMenuBar:)`, rebuilt on menu-open via `NSMenuDelegate`): resident + preference on shows the split "Close All Windows" (⌘Q, soft quit) and "Quit Kernova" (⌥⌘Q, true quit); the test host or preference-off shows a single "Quit Kernova" (⌘Q) that really quits. **Scripted/programmatic quits are honored** (#624): `classifyQuit` routes the Dock and the self-PID to `.stayResident`, but any other live sender — `osascript`, Script Editor, an updater, or a live-but-unidentifiable sender — to `.terminateAndSave` (a programmatic quit is explicit, so the accidental-⌘Q protection doesn't apply), alongside `loginwindow` and the unattributable fail-safe. A `.kernova` document-open reaches the single running instance directly through `application(_:open:)` (no launcher→XPC forwarding — the old #439 hack is gone). **"Open at Login" is opt-in:** a General-settings toggle over `SMAppService.mainApp` (`LoginItemService`), read live from `.status` and never persisted locally; it is the only thing that keeps the app resident across reboots.

`AppDelegate` is the entry point. It creates the `VMLibraryViewModel` and `VMLifecycleCoordinator`, opens the main window (test host) or starts headless and lets the cold-launch resolution decide (resident app), and manages the lifecycle of all windows. It tracks window controllers in dictionaries keyed by VM UUID, enabling one-to-many relationships (a VM can have a main view, a fullscreen window, and a clipboard window open simultaneously). `AppDelegate` also handles:

The View menu hosts the toolbar commands — "Show/Hide Toolbar" (⌥⌘T, `NSWindow.toggleToolbarShown(_:)`) and "Customize Toolbar…" (`NSWindow.runToolbarCustomizationPalette(_:)`) — as nil-target standard actions that resolve against the key window, so AppKit retitles/validates them without any `validateMenuItem` involvement; AppKit augments the View menu with its automatic window-tabbing items, while fullscreen/tiling commands land in the Window menu's AppKit-populated section (#318, #319)
- The Settings window (`Settings…` / ⌘,): `showSettings(_:)` lazily creates and retains a `SettingsWindowController` singleton
- Advanced lifecycle actions — "Start in Recovery Mode" and "Force Stop" — follow a uniform surfacing rule: **always visible in the Virtual Machine menu bar** (the full discoverability surface, where `validateMenuItem(_:)` greys out what doesn't apply), and **Option-revealed in the sidebar context menu** (gated by `AppPreferences.alwaysShowAdvancedOptions`, which shows them plainly when on). The preference governs only the sidebar reveal, not the menu bar.
- "Start in Recovery Mode" — gated on `VMInstance.canStartInRecovery` (stopped macOS guests). Menu bar: an always-visible item at ⌥⌘R. Sidebar: the Option-alternate of "Start". Selecting it routes through `VMLibraryViewModel.confirmStartInRecovery` → a `DetailAlertsPresenter` confirmation → `startInRecoveryConfirmed`
- "Rename" (no ellipsis — it starts an inline edit like Finder's single-item Rename, not a dialog) — reveals the sidebar surface, then routes to its inline rename: the handler fronts the library window (`showLibraryWindow`), uncollapses the sidebar (`MainWindowController.revealSidebar`), and calls `VMLibraryViewModel.renameVMInSidebar`. The reveal step is load-bearing — the settings pane’s Name field sits hidden behind a running VM’s live display (the original #320 no-op), and the sidebar itself can be collapsed or in a non-key window, so no surface is *unconditionally* visible; the command makes its target visible rather than assuming it. The settings Name field remains the click-to-rename affordance for the detail surface itself
- Lifecycle labels are status-dependent and shared across all three surfaces (menu bar, sidebar context menu, toolbar) via `VMInstance` display helpers: `startAction` ("Start" / "Install" / "Resume Install" per pending-install state) and `stopActionMenuTitle` / `stopActionToolbarLabel` ("Stop", or "Discard Saved State…" for a cold-paused VM, naming the destructive consequence — #321). The menu bar retitles in `validateMenuItem(_:)`; the toolbar in `VMToolbarManager.updateLifecycleGroup`
- "Show in Finder" — reveals the VM bundle via `NSWorkspace.activateFileViewerSelecting`. Present in both the menu bar and the sidebar context menu, and (uniquely among VM menu-bar actions) stays enabled while the VM is preparing, matching the sidebar's preparing menu — the bundle already exists on disk
- "Open VMs Folder" (File menu) — opens the VMs storage folder (`VMStorageService.vmsDirectory`, which creates it when missing) as a Finder window via `NSWorkspace.open`. VM-independent and always enabled, including while a VM is preparing and with no VM selected. Titled "Open … Folder" (a window of the folder's contents, like the Script menu's "Open Scripts Folder") rather than "Show in Finder" (reveal-selected-in-parent), and "VMs Folder" rather than "Library", which the Window menu already uses for the main window (#333)
- "Force Stop" — gated on `VMInstance.canForceStop` (`VMStatus.canForceStop` minus cold-paused, where there is no live VM to terminate and the retitled stop item — "Discard Saved State…" — is the single surface for that action). Menu bar: an always-visible sibling of "Stop". Sidebar: when a VM is gracefully stoppable (`VMInstance.canStop`) it is the Option-alternate of "Stop" (no top anchor needed — a Pause/Resume item always precedes the Stop pair); for transient states where graceful stop is unavailable (`canForceStop && !canStop`: starting/saving/restoring) it appears as a standalone, always-visible item. The always-visible menu-bar treatment is also load-bearing here: `Stop` is disabled during those transient states while `Force Stop` is the only working action, so a static ⌥-alternate would bury it. Selecting it routes through `VMLibraryViewModel.confirmForceStop` → a `DetailAlertsPresenter` confirmation → `forceStopConfirmed`
- "Move to Trash…" / "Delete Immediately…" — both gated on `VMInstance.status.canEditSettings`, both routing through `VMLibraryViewModel.confirmDelete(_:permanently:)` → the unified delete sheet (immediate sets `permanently: true`). "Delete Immediately…" is the Option-alternate of "Move to Trash…" in **both** surfaces: the sidebar context menu (gated by `alwaysShowAdvancedOptions` like the other advanced alternates) **and** the menu bar (⌘⌫ → ⌥⌘⌫). The menu bar is a deliberate exception to the "advanced actions are always visible in the menu bar" rule above (#276): unlike Recovery/Force Stop, an immediate delete is irreversible, so it follows Finder's File-menu idiom of hiding "Delete Immediately…" behind ⌥ rather than enumerating it always. Both menu items carry the HIG ellipsis (they open the input-gathering sheet); the buttons inside the sheet and the toolbar "Move to Trash" label do not (alert-button / toolbar-label convention)
- Suspend-on-quit behavior (suspending VMs before termination)
- Conditional termination: in the **resident app**, `applicationShouldTerminateAfterLastWindowClosed` always returns `false` rather than dropping to `.accessory` unconditionally — the global `willClose` observer (see above) already schedules that same window's activation-policy reconcile, which morphs to `.accessory` only if no user window remains, so the app stays resident and VMs keep running headless. In the test host it returns `false` only when VMs are active or fullscreen windows exist, preventing premature exit on fullscreen-to-inline transitions
- Dock icon reopen: in the resident app, `applicationShouldHandleReopen` summons the GUI (`.accessory`→`.regular` morph + show window). In the test host it restores the main window when clicked with no visible windows, or when clicked while the app is already active and the library window has been closed
- Fullscreen exit recovery: closing a fullscreen window automatically re-shows the main library window via `showLibrary(_:)`
- TCC relaunch: when macOS force-quits the app for a TCC permission change while VMs are running, `applicationShouldTerminate` launches `KernovaRelaunchHelper` (a CLI embedded in `Contents/MacOS/`) before beginning the async VM save. The helper monitors the app's PID via `DispatchSource` and relaunches the app via `NSWorkspace` after it exits, working around macOS's TCC relaunch timeout. It runs sandboxed (`app-sandbox` + `inherit`, adopting the app's sandbox at exec) and has no exec fallback — `NSWorkspace` is its only relaunch path (#89). TCC is positively identified by intercepting the `kAEQuitApplication` Apple Event and checking if the sender is the Privacy & Security settings extension (`com.apple.settings.PrivacySecurity.extension`).

`MainWindowController` creates an `NSWindow` with an `NSSplitViewController` as the content view controller. The split view has two panes: a sidebar (`NSSplitViewItem(sidebarWithViewController:)` wrapping the pure-AppKit `SidebarViewController` — a source-list `NSOutlineView`) and a detail pane (`DetailContainerViewController`, which hosts the AppKit empty-state ⇆ `VMDetailRouterViewController`). An `NSToolbar` with native `NSToolbarItem`s provides lifecycle controls (Start/Resume/Install, Pause, Stop — relabeled "Discard Saved State" for a cold-paused VM), Suspend, Clipboard, Pop Out, Fullscreen, New VM, and a Show/Hide Settings toggle that lets the user view the (read-only) settings form while a VM is running. Shared toolbar items (lifecycle, suspend, clipboard, pop-out, fullscreen, settings toggle) are managed by `VMToolbarManager`; the New VM button, the palette-only items, and sidebar items remain controller-specific. The settings toggle only appears in the main window — the pop-out display window passes `settingsToggleID: nil` — while the clipboard item appears in both windows (its nil-target `showClipboard` action targets the key display window's VM via `AppDelegate.activeInstance`). The main window's customize palette additionally offers Clone, Show in Finder, and Move to Trash as palette-only items (not in the default set) — VM-scoped verbs only; app-global commands like "Open VMs Folder" stay menu-bar-only — whose enablement mirrors the menu bar's `validateMenuItem(_:)` predicates via `NSToolbarItemValidation`. Toolbar state and the window title are observed via `withObservationTracking` (one `observeRecurring` loop, `observeWindowState`), and items are validated through `NSToolbarItemValidation`. The title tracks the selection — "Kernova — \<VM name\>", plain "Kernova" with none — so the active VM stays identifiable when the sidebar is collapsed; it also follows renames, since `VMInstance.name` is observable. Both the main and display toolbars are user-customizable (`allowsUserCustomization` + `autosavesConfiguration`, surfaced via the View menu's "Customize Toolbar…" — #319); the sidebar toggle and tracking separator are pinned via `toolbarImmovableItemIdentifiers`. The default layout right-aligns New VM and the toggle against the sidebar's trailing edge with a leading flexible space. While the sidebar is collapsed, New VM is removed from the toolbar (Safari's New Tab Group pattern) and restored on expand — `MainWindowController.syncNewVMVisibilityToSidebarState`, driven by KVO on the sidebar split item's `isCollapsed`, with autosave suspended around the removal (but live on the restore, which heals a foreign autosave that captured the collapsed layout), the removal's index mirrored into `AppPreferences.mainToolbarNewVMCollapseIndex` so it survives a relaunch, and the customize palette always presented with the canonical layout (mechanism details in [docs/TOOLBAR.md](TOOLBAR.md)). The `.fullSizeContentView` style mask and `.sidebarTrackingSeparator` preserve the full-height sidebar appearance matching Mail/Finder. The split view controller is a `SnapToFitSplitViewController` subclass that overrides `constrainSplitPosition` to magnetically snap the sidebar divider to the width that fully shows the longest VM name (computed by `SidebarViewController.widthToFitLongestRow()` from `SidebarVMRowCellView.contentWidth(forName:showsAgentAccessory:)`), clamped to the sidebar's `minimumThickness`/`maximumThickness` — the way Finder snaps its sidebar to the longest item.

### Models

**Files:** `VMConfiguration.swift`, `VMInstance.swift`, `VMBundleLayout.swift`, `StorageDisk.swift`, `USBDeviceInfo.swift`, `VMStatus.swift`, `VMBootMode.swift`, `VMGuestOS.swift`, `MacOSInstallState.swift`, `MacOSInstallContext.swift`, `KernovaUTType.swift`

The model layer has two key types:

- **`VMConfiguration`** is the persisted identity of a VM. It's a `Codable` + `Sendable` struct written as `config.json` inside each VM bundle. It holds: name, UUID, guest OS type, boot mode, CPU/memory/disk settings, display configuration (including `lastFullscreenDisplayID` for remembering which display a VM was fullscreen on), network settings, audio settings (`audioInputEnabled` — opt-in host microphone passthrough, defaults to off; `audioOutputEnabled` — guest audio routed to the host's audio output, defaults to on), guest agent state (`agentLogForwardingEnabled` for the opt-in vsock log fan-in, and `lastSeenAgentVersion` — the most recent guest-reported `Hello.agent_info.agent_version`, used to render a reconnecting agent as `.connecting` rather than the install nudge and to arm the post-start "didn't reconnect" watchdog), and OS-specific fields (macOS hardware model data, Linux kernel/initrd/cmdline paths).

- **`VMInstance`** is the runtime representation. It's an `@Observable` `@MainActor` class that wraps a `VMConfiguration`, an optional `VZVirtualMachine`, and a `VMStatus`. It references the VM's bundle path and provides computed properties for disk image, aux storage, and save file locations via `VMBundleLayout`. A view-layer extension (`VMInstance+Display.swift`) provides display properties (`statusDisplayName`, `statusDisplayColor`, `statusToolTip`) that distinguish preparing VMs (shown as "Cloning…"/"Importing…" in orange with a spinner), cold-paused VMs (state saved to disk, shown as "Suspended" in orange), and live-paused VMs (in memory, shown as "Paused" in yellow), plus the shared lifecycle action labels (`startAction`, `stopActionMenuTitle`, `stopActionToolbarLabel`) consumed by the menu bar, sidebar context menu, and toolbar so every surface titles the same action identically. The `PreparingOperation` enum (`.cloning`, `.importing`) provides display labels, cancel labels, and alert titles for preparing states. The `PreparingState` struct bundles the operation and its cancellable task into a single optional (`preparingState`) — when non-nil the instance is preparing, and `isPreparing` is a computed convenience. A per-instance `detailPaneMode` (enum `DetailPaneMode { case display, settings }`, defaulting to `.display`) lets the user toggle, while the VM is running, between the live display and a read-only view of the settings form; the mode is ignored when the VM is stopped (settings are always shown then). For macOS guests, `VMInstance` also owns a one-shot post-start watchdog: `startAgentPostStartWatchdog(grace:)` (kicked from `VirtualizationService.start` once status reaches `.running`) waits for the grace period (default 120 s) and flips `agentExpectedButMissing = true` if no guest agent `Hello` arrived in that window; the flag is cleared by `recordObservedAgentVersion(_:)` (called via the `VsockControlService` `onAgentVersionObserved` callback) and by `tearDownSession`. `VMInstance.agentStatus` synthesizes the `.expectedMissing(expected:)` case from `agentExpectedButMissing` + `configuration.lastSeenAgentVersion` so the rest of the UI sees the case alongside live `VsockControlService`-sourced states. Configuration mutations driven by guest activity (e.g. persisting a new `lastSeenAgentVersion`) flow through the `onUpdateConfiguration` closure, which `VMLibraryViewModel` wires to its centralized `updateConfiguration(of:mutate:)` dispatcher at every instance-construction site — so guest-driven and user-driven mutations share one persist + apply-live-policy path. `VMInstance` also exposes an `onAgentBecameCurrent` closure (wired the same way in `wirePersistence(for:)`) that `recordObservedAgentVersion(_:)` fires when a newly-observed agent version is current — i.e. an install/update just completed — driving the host-side auto-eject of the guest-agent installer disk via `VMLibraryViewModel.unmountGuestAgentInstaller`, so it runs regardless of which (if any) window is open. The runtime list of currently-attached USB mass storage devices (whether sourced from `removableMedia` at cold start or from later XHCI hot-attaches) lives on `VMInstance.liveRemovableMedia: [USBDeviceInfo]`, which the reconcile flow keeps in sync with the actual VZ device list.

`VMBundleLayout` is a `Sendable` struct that takes a bundle root path and provides all derived file paths (disk image, aux storage, save file, serial log, etc.), keeping path logic centralized.

The remaining models are enums: `VMStatus` (stopped/starting/running/paused/saving/restoring/installing/error), `VMBootMode` (macOS/efi/linuxKernel), `VMGuestOS` (macOS/linux), and `MacOSInstallState` (tracking download and install phases with progress). `VMStatus` provides computed properties for state checks (`canStart`, `canStop`, `canForceStop`, `canPause`, `canResume`, `canSave`, `canEditSettings`, `canRename`, `isTransitioning`, `isActive`). `canForceStop` covers all states where a `VZVirtualMachine` may exist and need forceful termination (running, paused, starting, saving, restoring); the instance-level `VMInstance.canForceStop` additionally excludes cold-paused (nothing in memory to terminate — discarding the saved state is that state's equivalent action), mirroring the `canStop`/`canSave` instance-level refinements.

### Services

**Files:** `ConfigurationBuilder.swift`, `VirtualizationService.swift`, `VMStorageService.swift`, `DiskImageService.swift`, `MacOSInstallService.swift`, `IPSWService.swift`, `USBDeviceService.swift`, `SystemSleepWatcher.swift`, `SpiceAgentProtocol.swift`, `AgentStatus.swift`, `ClipboardServicing.swift`, `ClipboardTransferIssue.swift`, `ClipboardTransferProgress.swift`, `ClipboardTransferProgressTracker.swift`, `SpiceClipboardService.swift`, `VsockClipboardService.swift`, `HostClipboardPublisher.swift`, `ClipboardPassthroughCoordinator.swift`, `HostClipboardFileProvider.swift`, `LoginItemService.swift`, `VsockControlService.swift`, `KernovaMacOSAgentInfo.swift`, `VsockListenerHost.swift`, `VsockGuestLogService.swift`, `VsockPorts.swift`, `AttachmentFileMonitor.swift`, `RuntimeFileAccess.swift`, `SerialSocketRelay.swift` (the macOS clipboard streaming engine — `ClipboardStreamSender`, `ClipboardStreamReceiver`, `ClipboardStreamTuning`, `KernovaCapability`, `AtomicGeneration`, `ClipboardTransferID` — lives in the shared `KernovaKit` package)

**Protocols:** `VirtualizationProviding`, `VMStorageProviding`, `DiskImageProviding`, `MacOSInstallProviding`, `IPSWProviding`, `USBDeviceProviding`

Services are split by concurrency requirements:

- **`@MainActor` services** (interact with `VZVirtualMachine`):
  - `VirtualizationService` — start, stop, pause, resume, save state, restore state. `start(_:)` has two branches: restore from a save file, or cold-boot (build a fresh `VZVirtualMachineConfiguration` and attach a new VM). The cold-boot path retries with bounded escalating backoff (`fileLockRetryDelay`, ~3.75 s budget) when the start fails on VZ file-lock contention, classified by `isFileLockContention`: `.invalidVirtualMachineConfiguration` carrying POSIX `EAGAIN` underneath (the "Failed to lock auxiliary storage" error, or its disk-image sibling) — matched by domain/code, never by localized text. The lock is taken on exactly the files the configuration builder attaches, so contention usually arrives wrapped in a `ConfigurationBuilderError` attach failure (which adds the item's identity for the removal offer); `isFileLockContention` unwraps via `underlyingAttachError` before matching, and `isTransientStartError` runs that check *ahead* of its "builder errors are permanent" rule — otherwise the wrap would silently disable both the retry and the transient classification. A previous `VZVirtualMachine` on the same bundle releases its advisory file locks only at *deallocation*, which lags `vm.state == .stopped` in proportion to guest memory size, so the post-install auto-boot (and a rapid manual stop→start) can transiently collide with the dying VM — see `MacOSInstallService` below for the state-gating half of this story. Lock contention is also a transient start error (`isTransientStartError`), so even an exhausted retry leaves the VM `.stopped` (grey, retryable) rather than `.error`.
  - `MacOSInstallService` — loads restore image, creates platform files (aux storage, hardware model, machine identifier), runs `VZMacOSInstaller` with KVO progress tracking. It **resolves the restore-image path through symlinks before either VZ hand-off** (`resolveRestoreImage`, via `PathValidation`): VZ does not resolve symlinks and rejects a path containing one in any component — `VZMacOSRestoreImage.load` reports `VZErrorInvalidRestoreImage` ("The restore image does not exist") while `FileManager` says the file is present and readable, and `VZMacOSInstaller.init(virtualMachine:restoringFromImageAt:)` is documented to raise an exception rather than throw. This is not defensive: under the sandbox the download destination is always symlinked, because `.downloadsDirectory` is the container's `Downloads`, itself a symlink to the real `~/Downloads` (#558). Same rule as `ConfigurationBuilder`'s storage attachments — VZ is only ever handed resolved URLs. After `installer.install()` resolves, it explicitly **waits for `vm.state` to reach `.stopped`** before returning. VZ's `install` completion handler fires while the post-install guest shutdown is still propagating through the framework's state machine — without the wait, the caller's auto-boot would cold-rebuild a `VZMacAuxiliaryStorage(contentsOf:)` while the install-side instance still held the file lock, producing the "Failed to lock auxiliary storage" error. Waiting also gives our `VZVirtualMachineDelegate.guestDidStop` a chance to fire, which releases our refs via `resetToStopped`; if the delegate doesn't fire within the timeout the install service tears down explicitly as belt-and-braces. The state gate is necessary but not sufficient: the advisory lock is released at the install-time VM's *deallocation*, which lags `.stopped` (more guest RAM → longer teardown — a 32 GB guest reliably outlasts the gate on Apple Silicon), and VZ offers no API to observe the release — `VirtualizationService`'s cold-boot retry (above) covers that residual window.

- **`Sendable` struct services** (no mutable state, safe to call from anywhere):
  - `VMStorageService` — creates/deletes/lists VM bundle directories at `~/Library/Application Support/Kernova/VMs/` and handles cloning (deep copy with new UUID). Deletion has two dispositions: `deleteVMBundle` (move to Trash via `trashItem`) and `permanentlyDeleteVMBundle` (immediate `removeItem`, the user-confirmed bypass-Trash path)
  - `DiskImageService` — creates ASIF disk images by decompressing bundled lzfse-compressed templates (sandbox-safe, no subprocess)
  - `IPSWService` (`final class` for `URLSession` lifecycle) — fetches available macOS restore images from Apple's catalog and downloads IPSW files directly into a Finder-visible `.kernovadownload` bundle (`Info.plist` + `data` at the bundle root). Streams response chunks via a `URLSessionDataDelegate` bridged to an `AsyncThrowingStream<Data, Error>` (the per-byte overhead of `URLSession.bytes(for:)` is unacceptable at multi-GB scale); manual HTTP `Range` / `If-Range` against the IPSW CDN drives resume, with the data file's on-disk size as the resume offset. On completion the `data` file is moved to the user-chosen `.ipsw` destination; disposing of the spent bundle is a separate, non-fatal step (`discardResumeData`), because the move is the download's success condition and a Trash failure must not undo it. The husk that a failed disposal leaves behind (`data` gone, `Info.plist` still present) is therefore a tolerated state, and every resume decision asks `IPSWBundle.isResumable` — which requires the `data` file, not just the directory — so a husk never reads as resumable or keeps the skip-existing fast path from firing. Cancellation safety: `streamBytes` re-checks `Task.checkCancellation()` **after** the for-await loop, because `AsyncThrowingStream.next()` resolves to `nil` (not throw) when the consumer's task is cancelled while parked — without that post-loop check, a user-cancelled download would proceed past the loop and finalize partial bytes onto the destination. A companion byte-count check (`totalWritten == expectedTotal`) catches the rarer "server closed cleanly under Content-Length" case. On any throw from this region the bundle is preserved so the next Start can resume from the partial bytes. Stale bundles are auto-discarded when the stored `originalURL` differs from the caller's request or when `Info.plist` fails to decode. The `.kernovadownload` UTI conforms to `com.apple.package` so Finder shows the bundle as a single icon

- **`SystemSleepWatcher`** — `@MainActor` observer class that monitors `NSWorkspace.willSleepNotification` and `NSWorkspace.didWakeNotification`. Follows the same pattern as `VMDirectoryWatcher`: callback-driven, `nonisolated(unsafe)` for observer tokens, `start()`/`deinit` lifecycle. Owned by `VMLibraryViewModel`, which uses it to auto-pause running VMs before sleep and resume them on wake.

- **`ConfigurationBuilder`** — Translates a `VMConfiguration` into a `VZVirtualMachineConfiguration`. Handles three boot paths: `VZMacOSBootLoader` (macOS), `VZEFIBootLoader` (EFI/UEFI), and `VZLinuxBootLoader` (direct kernel boot). Configures CPU, memory, storage, network, display, keyboard, trackpad, and audio devices. The virtio sound device carries an input stream when `audioInputEnabled` and an output stream when `audioOutputEnabled`; when both are off the sound device is omitted entirely. When `clipboardSharingEnabled` is set on a Linux guest, configures a `VZVirtioConsoleDeviceConfiguration` with a SPICE-named port using raw `VZFileHandleSerialPortAttachment` pipes (not `VZSpiceAgentPortAttachment`); macOS guests instead carry clipboard over the vsock device. For macOS guests it always appends a `VZVirtioSocketDeviceConfiguration` so the host can install vsock listeners (log + clipboard) against the live device once the VM is created. Resolves symlinks on user-supplied paths (shared directories, kernel/initrd, ISO images) and validates them before passing to VZ. File paths (kernel, initrd, ISO) are checked for existence and rejected if they point to directories. Shared directory validation checks existence, is-directory, readability, and writability (for read-write shares) against the resolved path.

- **`SpiceAgentProtocol`** — Pure data types and parsing for the SPICE agent wire format. Defines VDI chunk headers, VDAgent message headers, clipboard message types, and capability bitmasks. Includes `SpiceMessageBuilder` (builds wire-ready messages with a `port` parameter defaulting to `serverPort` for host-side use) and `SpiceAgentParser` (incremental parser handling fragmented data across multiple pipe reads). Fully `Sendable`, no I/O. Used only by the host-side `SpiceClipboardService` (Linux clipboard transport) — macOS guests now use vsock instead, so the file is no longer multi-target.

- **`AgentStatus`** — Enum (`.waiting | .current(version) | .outdated(installed, bundled) | .unresponsive(version) | .expectedMissing(expected) | .connecting(expected)`) that drives install/update/unresponsive/reinstall affordances in the sidebar and clipboard window, plus the Virtual Machine-menu guest-agent item's purpose-framed title (Install/Update/Reinstall/Manage) and attach-vs-eject mode. Sourced from `VsockControlService` for macOS guests (independent of clipboard sharing) and from `SpiceClipboardService` for Linux guests (`.waiting` / `.current` only). The UI reads it via `VMInstance.agentStatus`, which dispatches by `configuration.guestOS` and synthesizes `.expectedMissing(expected:)` / `.connecting(expected:)` from the post-start watchdog flag plus `configuration.lastSeenAgentVersion` — `VsockControlService` itself does not (and cannot) produce those cases because it has no access to persisted host state. The static `isObservedVersionCurrent(_:bundled:)` helper is the single source of truth for current-vs-outdated classification, shared by `VsockControlService.agentStatus` and the host-side auto-eject decision.

- **`ClipboardServicing`** — `@MainActor` protocol covering the public surface (`clipboardContent` — a `ClipboardContent` of ordered UTI-tagged representations, `isConnected`, `supportsBinaryRepresentations`, `lastTransferIssue`, `transferProgress` (a `ClipboardTransferProgress?` defaulting to `nil` via an extension, so text/eager transports report no progress), `inboundOfferSeq` (a `UInt64` bumped once per *new* inbound guest offer — not on our own writes or per-rep materialization — defaulting to `0` via an extension so fakes report none; the passthrough coordinator observes it to auto-publish exactly once per offer), `stop()`, `grabIfChanged()`, plus the lazy-receive hooks `materializeForPreview() async` and `materializeForCopy() async -> ClipboardContent`) shared by both clipboard implementations. The two materialize hooks have default no-op extension impls (return `clipboardContent` unchanged) so eager transports inherit them; only `VsockClipboardService` overrides them to pull lazily-offered guest representations on demand. `VMInstance.clipboardService` holds the existential, so the clipboard window controllers don't branch on transport. Agent install/version state is *not* part of this protocol — for macOS guests it lives on `VsockControlService` (the always-on control channel, independent of whether clipboard sharing is enabled). `SpiceClipboardService` exposes its own `agentStatus` directly for Linux.

- **`KernovaMacOSAgentInfo`** — Static accessors for the bundled guest agent. The "Package Guest Agent DMG" build phase extracts `CFBundleShortVersionString` from the built `Kernova Guest Agent.app/Contents/Info.plist` (via `plutil -extract`) and writes it to `Resources/KernovaMacOSAgentVersion.txt`; this file is the single source of truth for "what version did I bundle" and cannot drift from the actual agent binary. The accessor reads the sidecar lazily; missing/empty files trip an `assertionFailure` since the build phase is required to produce them. Also exposes the installer DMG URL.

- **`SpiceClipboardService`** — `@MainActor` `ClipboardServicing` implementation for Linux guests. Reads from the guest pipe on a background GCD queue, parses messages via `SpiceAgentParser`, and exposes `clipboardContent` (observable, editable by the UI) and `isConnected`. Text-only transport: only the UTF-8 text representation crosses the wire, and `grabIfChanged()` is a no-op for content without one (`supportsBinaryRepresentations == false`, so the window's intake explains rather than silently accepting; rich SPICE formats are tracked as an issue #112 follow-up). When the clipboard window loses focus, `grabIfChanged()` sends a `CLIPBOARD_GRAB` if the text was edited. Uses raw pipes rather than `VZSpiceAgentPortAttachment` so clipboard data flows through the gated UI instead of the host `NSPasteboard`.

- **`VsockClipboardService`** — `@MainActor` `ClipboardServicing` implementation for macOS guests, layered on `VsockChannel`, driving the chunk-streamed clipboard protocol via `ClipboardStreamSender` / `ClipboardStreamReceiver` (in `KernovaKit`). An outbound clipboard change is announced as a **metadata-only** `ClipboardOffer` carrying a `ClipboardRepresentationInfo` per representation (`uti`, `byteCount`, `filename`, `isInline`, `isDirectory`) under a monotonically increasing `generation` — no bytes leave at offer time. The offer also carries an offer-level `is_concealed` flag (set when the source pasteboard marked the snapshot `org.nspasteboard.ConcealedType`); a concealed inbound offer publishes concealed content that the window hides behind a placeholder and `materializeForPreview()` skips (the secret is never pulled into host memory for a preview), though `materializeForCopy()` still streams it on an explicit Copy-to-Mac. **Inbound is a lazy hybrid.** `handleOffer` publishes a metadata-only `.pendingRemote` placeholder per offered rep to `clipboardContent` immediately (an `InboundPromise` holds the offered rep metadata, in offer order, plus a per-rep materialization cache) and pulls **nothing**; the clipboard window pulls only the reps it renders richly — text, inline RTF, and images up to `ClipboardPreviewPolicy.maxEagerPreviewBytes` (32 MiB) — **on display** via `materializeForPreview()` (idempotent per generation, started from `viewDidAppear`/status-update when the window is visible), and `materializeForCopy()` pulls every remaining rep when the user picks "Copy to Mac". Each rep is pulled at most once (cached in the promise). A single `pull(...)` bridges the off-actor stream receive to `async`/`await` via a `CheckedContinuation` resumed by the receiver's per-`transfer_id` `awaitTransfer` hook — wrapped in a single-resume `PullContinuation` box, since the off-main awaiter, the on-main send-failure catch, and a `lazyPullTimeout` backstop all race a channel teardown (the first resume wins, the rest are no-ops; the backstop also resolves a pull the guest accepts but never streams while the channel stays open); it answers each request with a `ClipboardStreamBegin` → `ClipboardChunk`* → `ClipboardStreamEnd` sequence under windowed flow control (`ClipboardStreamAck` advances the credit window; `ClipboardStreamAbort` tears a transfer down), with a streaming SHA-256 verified at End. A small inline representation reassembles in RAM; a large inline one (past `maxResidentInlineBytes`) or a file representation streams to a temp file under the receiver's free-space guard, never resident whole — a spilled inline rep is served back via a memory-mapped read so inline content has no size cap, and large files never cross the wire until the user copies them. `grabIfChanged()` refuses to offer content that still holds `.pendingRemote` placeholders back to the guest (the sender can't stream a placeholder and received content shouldn't echo). A **copied file** (outbound) is offered from a `stat` only — a disk-backed `.file` representation that is never read at offer time and has no size cap — and streamed on request. **Multiple files** ride as several `.file` reps in one offer (`ClipboardContent` partitions reps by filename: each non-empty filename is a distinct file, the rest are one inline item); the inbound side materializes each and the window writes one `NSPasteboardItem` per file (`hostPasteboardItems`), so a Finder paste creates N files. A **folder** (or OS package) rides as one `.file` rep flagged `isDirectory` whose bytes are an in-process AppleArchive (`ClipboardDirectoryArchive`, `.aar`/LZFSE) of the tree: the sender archives eagerly off the main actor into its staging (the offer needs the archive's size and the stream its SHA-256), `repInfo(for:)` carries the flag, and because the stream layer is offer-agnostic the receiver re-tags the delivered rep from the offer (`pull`) and the window extracts the `.aar` into a real folder (`stagedFileURL`, under a double-disk free-space guard) so a Finder paste recreates the tree. **With `clipboard.dirtree.v1` negotiated (folder D1b, #422) this eager-archive path is only the fallback:** a directory rep instead crosses as a `.directory` **source** rep (the producer retains the folder, no archive) carrying a stat-walk size *estimate*, and — when the File Provider toggle is on — routes at paste time as a placeholder *tree* (`ClipboardDirectoryTree` walks the source on demand: a `ClipboardTreeFetch` listing, then each child file over the SHA-verified stream; the host publishes it via `FileProviderPublishFolder` → the hierarchical `FileProviderManifest`, and child `fetchContents` relays a child-addressed pull). Only a capability-absent peer or a File-Provider-off paste hits the archive fallback, which now archives at **request** time (`archiveAndStream`), bounded by the deadline-safe total cap. Mixing files and folders in one copy just works — each selected item is its own rep. Same-named files in one copy get distinct staged URLs from `ClipboardFileStaging` (it uniquifies a repeated name with a ` (n)` suffix), and an offer is capped to `ClipboardContent.maxOfferableRepresentations` (0xFFFF) so a `transfer_id`'s 16-bit rep index can't alias. The eager `InboundCollection`/`onTransferComplete`/`commitInbound` assembly is retired; the channel-wide receiver `onComplete`/`onAbort` is now a log-only fallback for an unexpected unawaited transfer. Stale generations are detected by both sides (`AtomicGeneration` mirrors the current outbound generation for the sender's off-actor supersession check) so a transfer that races a newer offer aborts rather than overwriting fresher state. A `ClipboardRequest` the receiver can't answer — stale generation, out-of-range rep index, or UTI mismatch — is answered with a `ClipboardStreamAbort` (`ClipboardStreamSender.rejectRequest`) rather than silently dropped, so a peer's parked lazy pull wakes immediately instead of stalling to its `lazyPullTimeout` backstop (#357). Offer dedup and echo suppression compare 32-byte `ClipboardContent.digest` values, never retained copies (a `.file` rep folds in its streamed sha256; a `.pendingRemote` placeholder uses a distinct digest domain tag). A transfer the requester can't accept — or a volume that fills mid-stream — fails as a `.diskFull(needed:available:)` `ClipboardTransferIssue`, surfaced (like clipboard-scoped peer error frames) through the observable `lastTransferIssue` for the window's transient-message strip. **Live transfer progress** is surfaced in both directions (#354): the sender's per-chunk `onProgress`/`onComplete` and the receiver's `(bytesReceived, totalBytes)` `onProgress` (inbound recorded *through* the `PullContinuation` resume gate, so a chunk landing after the pull resolves can't resurrect the bar) feed a lock-guarded `ClipboardTransferProgressTracker` off-main; a conflation flag bounds the *queue depth* (at most one main-actor flush pending) while a per-transfer `FetchProgressCoalescer` — the same shared KernovaKit throttle the File Provider pull's two progress consumers use (~1% of total or ~100 ms apart, always the final chunk) — bounds the republish *rate*, so a 64 KiB-chunked pull can't drive a main-actor hop per chunk (#636), and a per-transfer reveal `Task` (default ~300 ms, injectable for tests) means a sub-delay transfer never flashes (an unrevealed transfer's chunks never schedule a flush at all, since they can't change the projection); the projected most-significant transfer is published on the `@MainActor` observable `transferProgress`, cleared on every terminal (the single post-`await` inbound clear-point, the outbound `onComplete`, and `stop()`). The service is constructed lazily by the vsock clipboard listener when the guest connects, so `VMInstance.clipboardService` may be nil until then. No `Hello` is exchanged on this channel — version/liveness and the `clipboard.stream.v1` capability gate live on the always-on control channel (`VsockControlService`); the service only runs when the peer advertised that capability, so there is no legacy single-frame fallback. The O(payload) digest work runs off the main actor via `ClipboardContent.makeOffActor`, and per-transfer chunk I/O runs on the sender's/receiver's own serial queues so a big copy/paste never blocks the UI; the host uses `@MainActor` `async`/`await` for its pulls (no blocking), while the guest agent blocks its main thread (woken off-main — see below). (Giant *inline*, non-file content still touches the main thread for the unavoidable `NSPasteboard` read/write.)

- **`VsockControlService`** — `@MainActor` `@Observable` consumer of the always-on control channel (`KernovaVsockPort.control = 49154`). Sends a host `Hello` on start; processes the guest's `Hello` to populate `agentVersion` and `isConnected`; emits and consumes bidirectional `Heartbeat` frames on a configurable cadence (default 5 s). A liveness watchdog flips `agentStatus` to `.unresponsive(version:)` after `unresponsiveAfter` (default 15 s) of inbound silence, and tears down the channel after `terminateAfter` (default 30 s) — letting the listener accept a fresh connection. `agentStatus` classifies the guest-reported `Hello.agent_info.agent_version` against `KernovaMacOSAgentInfo.bundledVersion` via `AgentStatus.isObservedVersionCurrent(_:bundled:)`, which delegates to `KernovaVersionComparison.isAtLeast` in the `KernovaKit` package — the single numeric comparator (`String.compare(_:options: .numeric)`, correct for dotted-decimal SemVer like `0.9.0` vs `0.10.0`) shared by the host's `agentStatus` / `VMInstance.recordObservedAgentVersion` auto-eject decision *and* the guest's own menu update line, so all sites agree on current-vs-outdated; a `bundledAgentVersion` initializer parameter lets tests vary the comparison target. The host's `Hello` also carries `bundled_agent_version` (the version this host ships) host→guest, so the guest can label its own update state; the guest leaves the field empty in its Hello and an empty value reads as "unknown" (no update prompt). The listener is installed unconditionally for every macOS guest with a `VZVirtioSocketDevice`, so `agentStatus` is meaningful even when clipboard sharing is disabled. The host `Hello` advertises `KernovaCapability.controlChannelDefaults` (control, heartbeat, and `clipboard.stream.v1`); the guest's `Hello` capability list is inspected and the `PolicyUpdate` clipboard bit is gated on the guest advertising `clipboard.stream.v1` (mirrored on the guest side) — an agent that can't stream keeps clipboard disabled and the host shows the existing `.outdated` "update the guest agent" affordance. Also delivers the per-VM toggle policy to the guest: an optional `policyProvider: () -> AgentPolicySnapshot` closure is read on every guest `Hello` and the resulting snapshot is sent as a `PolicyUpdate` frame so the guest agent knows which capabilities (log forwarding, clipboard sharing) to enable. `sendPolicyUpdate(_:)` is also public so runtime hot-toggle pushes can reuse the same path. An optional `onAgentVersionObserved: (String) -> Void` closure fires once per guest `Hello` carrying a non-empty `agent_version`; `VMInstance.startVsockServices` wires it to `recordObservedAgentVersion(_:)`, which persists the new value via `onUpdateConfiguration` (routed through the view-model's `updateConfiguration` dispatcher) and clears the post-start watchdog state. The completed-Hello state is also the per-VM admission anchor (#145): `VMInstance.admitsFeatureChannel` reads `isConnected` (and, for the clipboard port, `guestSupportsClipboardStreaming`) in the log/clipboard listeners' `shouldAdmit` predicates, so no feature channel is accepted for a VM whose agent hasn't completed the control handshake; peer-supplied `Hello.capabilities` strings are rendered via `KernovaCapability.logDescription(of:)` so a malicious peer can't write arbitrary content into the persisted log.

- **`AgentPolicySnapshot`** — Plain `Equatable` `Sendable` struct (`logForwardingEnabled`, `clipboardSharingEnabled`) carrying the pair of toggles the guest agent honors. Decouples `VsockControlService` from `VMConfiguration`: the host injects a closure that reads the latest config when policy is sent, so reconnects always pick up the current snapshot.

- **`VsockListenerHost`** — `@MainActor` wrapper around `VZVirtioSocketListener` bound to one vsock port. The nonisolated `VZVirtioSocketListenerDelegate` callback dups the connection's file descriptor and configures it (`configureAcceptedSocket`) before bridging back to MainActor to construct a `VsockChannel` and hand it to a caller-supplied closure. Two socket options are set on the accepted fd: `SO_SNDBUF` is raised to 1 MiB and `SO_SNDTIMEO` is set defensively (a belt-and-suspenders backstop; Apple does not document whether it is honored on a vsock fd). The `SO_SNDBUF` bump is the host→guest throughput lever (#377): a `VZVirtioSocketConnection`'s host-side fd is born at XNU's 8 KiB `net.local.stream.sendspace` default and vsock throughput is gated by the *writer's* send buffer, so 1 MiB lifts the host→guest **transport** ceiling ~9× (knee measured at 256 KiB), after which the app stack is the only host→guest limiter. It is host-only — the guest→host writer lives in Apple's `com.apple.Virtualization.VirtualMachine` helper process and is unreachable per-fd (so that direction stays capped ~750 MiB/s). The 1–2 MiB streaming credit window can still exceed the 1 MiB send buffer at its ceiling, so a `writeFramed` write can still block — what keeps that from wedging the channel is `VsockChannel`'s split write/state lock (#457), which lets inbound acks and control frames keep flowing while a write is parked, so the streaming engine's own no-ack/inbound-stall deadlines can still fire and tear the channel down. An optional `shouldAdmit` predicate is evaluated per accepted connection before any channel is built — returning `false` refuses the connection at the VZ level (#145); `VMInstance` wires the log/clipboard listeners to `admitsFeatureChannel`, which requires a control channel with a completed `Hello` (clipboard additionally requires the negotiated `clipboard.stream.v1`), while the control listener admits unconditionally (it *is* the handshake channel). One instance per service; multiple coexist on the same `VZVirtioSocketDevice`.

- **`VsockGuestLogService`** — `@MainActor` consumer that owns one accepted `VsockChannel` for the lifetime of a guest connection. Forwards `LogRecord` frames through a `GuestLogEmitter` abstraction (default `OSLogGuestLogEmitter`, subsystem `app.kernova.guest`); guest log levels map 1:1 onto `os.Logger` methods. `Error` frames go to the host's own diagnostic logger; any other payload on this port (`Hello` / `Heartbeat` / clipboard frames belong on their own channels) logs a wrong-port warning and closes the channel (#145) — a conformant agent's reconnect loop re-establishes it. The service self-terminates on EOF.

- **`VsockPorts`** — Central registry of port assignments (`KernovaVsockPort.control = 49154`, `KernovaVsockPort.clipboard = 49152`, `KernovaVsockPort.log = 49153`) so each service gets its own listener on a distinct port instead of in-band multiplexing. The control listener is always installed for macOS guests with a `VZVirtioSocketDevice`; the log listener is gated on `configuration.agentLogForwardingEnabled`; the clipboard listener is gated on `configuration.clipboardSharingEnabled`. Both gates are also re-evaluated at runtime via `VMInstance.applyLivePolicy(oldConfig:newConfig:)` — flipping the toggle while a macOS VM is running installs or tears down the listener and pushes a fresh `PolicyUpdate` to the guest agent. Linux clipboard sharing is restart-only (the SPICE port must be declared at config-build time).

- **`USBDeviceService`** — `@MainActor` implementation of `USBDeviceProviding`: runtime USB mass storage attach/detach against the live VM's XHCI controller, producing/consuming `USBDeviceInfo`. Owned by `VMLifecycleCoordinator`; backs the removable-media reconcile flow (below) and guest-agent installer mounting. Mocked by `MockUSBDeviceService` / `SuspendingMockUSBDeviceService`.

(Four more services — `HostClipboardFileProvider`, `LoginItemService`, `AttachmentFileMonitor`, and `SerialSocketRelay` — are described where they're most relevant: their Directory Structure entries above, the launch-model paragraph, and the Serial console data flow section.)

**Storage topology mirrors VZ.** `VMConfiguration` carries two ordered lists that map directly onto VZ's two storage surfaces. `storageDisks: [StorageDisk]?` maps onto `vzConfig.storageDevices`; position [0] boots first on EFI guests. Each entry's `kind` (`.virtio` or `.usbMassStorage`) is inferred from the file extension at add-time via `StorageDisk.defaultKind(forPath:)` — `.iso`/`.dmg` default to USB mass storage so installer media doesn't shift the main disk's `/dev/vda` letter when reordered for boot, everything else defaults to virtio. `removableMedia: [RemovableMediaItem]?` maps onto `usbControllers[0].usbDevices` — hot-pluggable, no boot semantics. The same bundled-disk entry (`Disk.asif`, internal, virtio) appears in `storageDisks` as a regular row; nothing in the data model singles it out as "the main disk." Each `StorageDisk` carries a user-editable `label` (renamed inline or via the row context menu through `VMLibraryViewModel.renameStorageDisk`; cosmetic — the backing file keeps its stable UUID name and the guest is unaffected); new disks get a collision-free default label via `StorageDisk.uniqueLabel`. The model stores **no** physical sizing — on-disk footprint, virtual capacity, and creation date are read live from the file at display time (uniformly for the main disk and additional disks), since a size can change out-of-band (e.g. a CLI resize) and any stored copy would go stale. The subtitle reads run **off the main thread** via `populateDiskSubtitle` (placeholder → detached read → fill-in; reads are cancelled when a field is re-bound); Get Info also reads off-main (a detached size + creation-date read, then presents). Both attachment lists share one `VMSettingsViewController` implementation — `refreshAttachmentList`/`makeAttachmentRow` (rebuild vs. in-place), one `buildAttachmentContextMenu` with eight `@objc` handlers dispatched on an `AttachmentRef(kind, id)` `representedObject` (Eject is removable-only, so its handler is a no-op for storage refs), `presentAttachmentInfoPopover`, and `commitAttachmentRename` — parameterized by `AttachmentKind` rather than duplicated per list. **Removable media (`RemovableMediaItem`) is at full parity with storage disks**: same editable `label` (inline rename or context menu, via `VMLibraryViewModel.renameRemovableMedia` — safe to rename while running, since a label-only edit leaves `path`/`readOnly` untouched and the reconcile only detaches/reattaches on those, so the medium stays mounted), the same `AttachmentRowView` UI **plus an inline trailing Eject button that storage rows omit** (a one-click detach — no confirmation, file untouched, via `removeRemovableMedia(_:from:trashFile: false)` — hot-pluggable media is swapped often, so it earns a dedicated control; the eject glyph is shared with the Shared Directories row button), the right-click menu (Rename / Get Info / Show in Finder / Copy Path / Copy File Name / Read Only / **Eject** / Remove — Eject, the no-confirmation detach, is removable-only, the one menu item storage rows lack), the same `AttachmentInfoPopoverContentViewController`, and the same live on-disk/allocated sizing (always external — a raw `.iso`/`.dmg` reports its logical size as the capacity).

**Live-editable fields and their dispatch.** `VMConfiguration.liveEditableFieldsChanged(old:new:)` is the single source of truth for "did anything change that should take effect while the VM is running?" It combines `hotToggleFields` (a typed `[KeyPath<VMConfiguration, Bool>]`) with `removableMediaChanged(old:new:)`, which array-compares the `removableMedia` lists. `storageDisks` changes are deliberately NOT live-editable — they go to `vzConfig.storageDevices`, which VZ requires fixed at start time, so the settings UI keeps that section locked while the VM is running. `VMSettingsViewController` mutates via the centralized `VMLibraryViewModel.updateConfiguration(of:mutate:)` dispatcher (persist + `applyLivePolicy`). Every control's target/action handler routes its write through this dispatcher; no settings control writes to `instance.configuration` directly.

**Removable-media reconcile.** `VMLibraryViewModel.applyLivePolicy(for:old:new:)` forks: vsock listener changes go through `VMInstance.applyLivePolicy`; `removableMedia` changes are dropped into `pendingRemovableMediaTarget` and a coalesce-and-drain task (`runRemovableMediaReconciliation`) calls `applyLiveRemovableMediaChange(for:target:)` until the pending dictionary empties. The reconciler computes a per-id diff against `instance.liveRemovableMedia` (add / remove / mutate-in-place / reorder-noop), detaches first to avoid duplicate-UUID conflicts on swaps, then attaches. `deviceNotFound` (guest-side ejection) and `noVirtualMachine` (VM torn down) are handled distinctly; on any other framework error the reconciler calls `reconcileConfigToLiveState(for:lookup:)` to rebuild `config.removableMedia` from `instance.liveRemovableMedia` — so the UI snaps to what's actually attached instead of describing a state VZ refused. The rollback bypasses `updateConfiguration` (direct write + `saveConfiguration`) to avoid re-entering the reconcile pipeline.

**Save-state device UUID persistence.** `VZUSBDeviceConfiguration.uuid` is matched against the saved-state file's recorded device list during `restoreMachineStateFrom(url:)` — fresh UUIDs each launch would break restore. `RemovableMediaItem.id` becomes `VZUSBMassStorageDeviceConfiguration.uuid` and is persisted with the entry in `config.json`. For virtio entries in `storageDisks`, the entry's `id` is also used as the `VZVirtioBlockDeviceConfiguration.blockDeviceIdentifier` (truncated to 20 ASCII chars), giving Linux guests a stable `/dev/disk/by-id/virtio-<identifier>` symlink — with the exception of the bundle's primary disk (`Disk.asif`, identified by `ConfigurationBuilder.isMainBundleDisk(_:layout:)`), which intentionally has no `blockDeviceIdentifier` set so the pre-refactor `/etc/fstab` behavior is preserved. The synthesized default main disk (used when `storageDisks` is nil/empty) derives its UUID deterministically from `SHA256(bundleURL.path)` via `ConfigurationBuilder.stableMainDiskID(forBundleAt:)`, so entry-by-id lookups (notably `removeStorageDisk` and the settings list's read-only/delete row actions) are stable before the user has materialized the list. `clonedForNewInstance` regenerates every `StorageDisk.id` and `RemovableMediaItem.id` so two bundles don't share device identity.

All service implementations conform to protocols defined in `Services/Protocols/`. This enables full dependency injection — tests use mock implementations that track call counts and support error injection.

### ViewModels

**Files:** `VMLibraryViewModel.swift`, `VMLibraryPresenting.swift`, `VMLifecycleCoordinator.swift`, `VMCreationViewModel.swift`, `VMDirectoryWatcher.swift`

- **`VMLibraryViewModel`** is the central `@Observable` view model. It owns the array of `VMInstance`s and handles list-level operations: add, remove, rename, reorder, selection tracking. VM order is user-customizable via drag-and-drop in the sidebar, persisted as `AppPreferences.vmOrder`; selection likewise persists via `AppPreferences.lastSelectedVMID` (both injected through `VMLibraryViewModel.init(preferences:)`, defaulting to `.shared`, #528). VMs not in the custom order (newly created/discovered) sort after ordered VMs by `createdAt`. For lifecycle operations (start, stop, install), it delegates to `VMLifecycleCoordinator`. Clone and import operations use a "phantom row" pattern: a `VMInstance` with `isPreparing = true` appears immediately in the sidebar with a spinner while the file copy runs asynchronously on a bounded background queue. Both operations share this lifecycle through two private helpers — `registerPhantom(_:)` (append/sort/persist/select the new row) and `prepareBundle(_:operation:copyWork:onSuccess:)` (spawns the copy `Task` and wires success/deallocated/cancelled/failure cleanup) — so the delicate weak-self/cleanup logic exists once rather than being duplicated per operation kind (#488). `reserveAndImport(from:)` reserves a collision-free destination and registers the bundle's phantom row **synchronously** — its collision check consults the reserved `bundleURL`s of in-flight phantoms in `instances`, not just on-disk `fileExists` (delegating the "Name / Name 2 / …" climb to the shared `UniqueName`, matched case-insensitively for APFS, #498) — then spawns the copy `Task`, which runs its blocking copy concurrently, bounded by a shared `copyQueue` so a large drop can't saturate the pool (#497), like clone's. `importVMs(fromDroppedURLs:)` — the single import entry point `AppDelegate` (Finder open) and `SidebarViewController` (drag-and-drop) both call — filters to `.kernova` and reserves every bundle in one synchronous pass. Because every phantom is registered before any `await`, a whole batch's destinations — and two overlapping triggers' (e.g. a drag-and-drop batch still copying bundle 3 of 5 when a Finder double-click of another bundle arrives) — are reserved atomically on the MainActor with no serialization: every bundle imports (#444), overlapping triggers never collide (#487), and an unrelated trigger's phantom appears immediately instead of waiting behind another batch's copies (#491). That atomicity is inherent to the reservation step rather than to any external ordering, so there is no serialization mechanism a future caller could bypass (#492/#493). `cloneVM(_:)` is likewise fire-and-forget — a clone's UUID-named destination bundle can never collide with another clone or an import. Neither import nor `cloneVM(_:)` gates on `hasPreparing` (the "operation in progress" reject alert was removed, #487) — the computed property now only drives the Clone command's disable state (`MainWindowController`, `AppDelegate` menu, `SidebarViewController` context menu) and the `reconcileWithDisk()` skip. Cancelling a preparing row (`cancelPreparingConfirmed`) cancels the task and marks the row **"Cancelling…"** but keeps it in `instances` until the uninterruptible copy settles; the copy `Task` is then the single owner of the settle — it removes the row and trashes the bundle exactly once, with no race and no phantom success logged, and the row's continued presence keeps the destination reconcile-protected meanwhile (#496). (If the copy already finished, cancel removes and trashes it synchronously.) The app-quit sweep (`AppDelegate.cancelAndCleanupPreparingInstances()`) cancels + trashes each preparing instance via `isPreparing` directly; because every accepted bundle registers its phantom synchronously and a cancelling row stays preparing, the sweep sees them all (#491/#496). Force-stop is surfaced via `confirmForceStop()` which presents a confirmation dialog. **All `VMConfiguration` mutations route through `updateConfiguration(of:mutate:)`** — a single dispatcher that takes an `inout` closure, no-ops when the closure produces an equal value, persists via `saveConfiguration`, and calls `applyLivePolicy(for:old:new:)`. This is the only place persist + live-policy fire together; settings-UI bindings (`configBinding`, `storageDiskBinding`, `removableMediaBinding`), install/uninstall flows, rename, display-window callbacks, and guest-driven `VMInstance.onUpdateConfiguration` writes all funnel through it.

- **`VMLifecycleCoordinator`** is an `@MainActor` coordinator that owns the lifecycle services (`VirtualizationService`, `MacOSInstallService`, `IPSWService`, `USBDeviceService`, plus the `FileSystemOperating` seam, which `VMLibraryViewModel` also holds for its own trash/remove call sites and passes down so both share one injected value); `VMStorageService` and `DiskImageService` are owned by `VMLibraryViewModel`. It orchestrates multi-step operations like macOS installation (which involves IPSW download → platform file creation → VM configuration → installation). This separation keeps `VMLibraryViewModel` focused on list management. The coordinator enforces **per-VM operation serialization** — at most one lifecycle operation can be in flight for a given VM at any time; concurrent requests are rejected with `VMLifecycleCoordinator.LifecycleError.operationInProgress`. `stop` and `forceStop` bypass serialization entirely (clearing the active-operation token before calling the service) so users can always cancel hung operations.

- **`VMCreationViewModel`** drives the multi-step creation wizard. It tracks the current step, validates inputs at each stage, and produces a `VMConfiguration` + disk image on completion. It is a pure `@Observable` state machine with no UI framework dependency — the wizard's pure-AppKit step view controllers (`Views/Creation/`) read/write it directly and the shell observes it via `observeRecurring`.

- **`VMDirectoryWatcher`** uses `DispatchSource.makeFileSystemObjectSource` to monitor the VMs directory for external changes (e.g., a user restoring a VM from Trash via Finder). When changes are detected, it triggers reconciliation in `VMLibraryViewModel` to sync the in-memory list with disk.

- **`SystemSleepWatcher`** (see Services section) is also owned by `VMLibraryViewModel`, triggering `pauseAllForSleep()` and `resumeAllAfterWake()` on system sleep/wake events. Auto-paused VMs are tracked in `sleepPausedInstanceIDs` so user-paused VMs are not accidentally resumed.

### Views

**Files:** the entire app target is pure AppKit — `NSViewController`/`NSView` subclasses plus free `make*` atom-factory functions, with no `import SwiftUI` anywhere. The detail pane (settings, router, install progress, placeholders, empty state) was the last SwiftUI surface and is now AppKit: `VMDetailRouterViewController` routes by status (via the unit-tested `DetailRoute.resolve`) to `VMSettingsViewController`, `DetailStatusPlaceholderViewController`, `MacOSInstallProgressViewController`, or the display placeholder. **Every popover, sheet, and alert is end-to-end AppKit**: missing-attachment (`AttachmentIconButton` → `MissingAttachmentPopoverContentViewController`), attachment Get Info (`AttachmentRowView` context menu → `AttachmentInfoPopoverContentViewController`, shared by storage disks and removable media), Storage Disk / Removable Media Create (`DiskSizePopoverCoordinator` → `DiskSizePopoverContentViewController`), the generic info-popover surface (`InfoButtonView` → `InfoPopoverContentViewController`), the mic-permission warning popover (`MicrophonePermissionPopoverContentViewController`), the Boot Order and Delete-VM sheets (via `SheetPresenter`), the lifecycle confirmation alerts (`DetailAlertsPresenter` → `presentSheetAlert`), and the sidebar agent-status popover (`SidebarVMRowCellView` → `SidebarAgentStatusButtonView` → `AgentStatusPopoverContentViewController`). Each uses a real `NSPopover`/`NSAlert`/sheet; controllers build their full layouts in `loadView()` using shared token sets (`CalloutStyle` for popovers, `GroupedFormStyle` for grouped forms shared with the wizard) + atom factory functions. **All popover anchors target a wrapper `NSView`** (never an inner control) so `NSPopover.preferredEdge` semantics are interpreted in an unflipped coordinate system. **No shared callout/form container or base class** — visual consistency comes from shared tokens, not inheritance. **Genuinely shareable controllers are reused via init parameterization** (`DiskSizePopoverContentViewController` serves both Create flows via `headline`/`caption`; `InfoPopoverContentViewController` takes `[InfoPopoverParagraph]` distinguishing `.body` from `.code`). AppKit content controllers decouple from `VMLibraryViewModel` via delegate protocols; their hosts (settings VC, alerts presenter) implement the delegates and forward user choices to the view model. Conversely, the view model surfaces alerts, sheets, and the wizard by calling its `VMLibraryPresenting` delegate (`DetailContainerViewController`) imperatively — not by toggling observed `show*` flags. Errors raised before the delegate attaches (e.g. the initial `loadVMs()` in `init`) are buffered on the view model and flushed when it is set.

Views observe `VMLibraryViewModel` and individual `VMInstance`s via the Observation framework (`observeRecurring`). The view hierarchy (AppKit throughout):


**Serial console data flow:** the guest serial output pipe has a single reader —
`VMInstance.startSerialReading`'s `readabilityHandler` (background GCD queue) — which
fans out to (a) `serial.log` (authoritative, always on) and (b) `SerialSocketRelay.forwardOutput`
(best-effort tee, only when `serialSocketRelayEnabled`). An external terminal attaches to the
relay's AF_UNIX socket; bytes it sends are written straight to the serial input pipe, making the
relay the sole host-side writer of serial input. There is no in-app terminal emulator and no serial
window — emulation is delegated to the user's terminal. The connect instructions and the (per-VM
deterministic) socket path are surfaced via the info button on the **Serial Console** section of
`VMSettingsViewController` (the path is `VMInstance.serialSocketPath(for:)`, computed the same way
the relay binds it). The relay socket lives under `NSTemporaryDirectory()` (short path; the bundle
dir overflows `sockaddr_un.sun_path`) and is App-Sandbox/Mac-App-Store-compatible (no entitlement).
The per-VM `serialSocketRelayEnabled` flag is hot-toggleable via `applyLivePolicy` →
`applyLiveSerialRelayPolicy`, handled before the vsock-socket-device guard since the relay is
host-only.

## Key Design Decisions

### 1. Pure-AppKit UI

**What:** The app is entirely AppKit. `NSSplitViewController` owns the sidebar/detail layout, `NSToolbar` with native `NSToolbarItem`s the toolbar, and `NSWindow` window management. The sidebar is a source-list `NSOutlineView` (`SidebarViewController`); the detail pane is an empty-state view ⇆ `VMDetailRouterViewController` (`VMSettingsViewController` + placeholders + install progress). The VM display (`VZVirtualMachineView`) is managed by `VMDisplayBackingView` — in the detail pane `DetailContainerViewController` layers it on top of the detail content, and in pop-out/fullscreen windows `VMDisplayWindowController` uses it directly as the window's content view. The app target imports no SwiftUI.

**Why:** The app needs precise control over native macOS chrome — toolbar items, split-view behavior, sidebar collapsibility, popover/sheet anchoring — and Swift 6 strict-concurrency-friendly observation. SwiftUI's `NavigationSplitView`/`.toolbar` and the AppKit↔SwiftUI bridge boundaries proved fragile for this; the project incrementally converted every surface (sidebar, wizard, popovers/alerts, then the detail pane) to AppKit. Toolbar state is validated via `NSToolbarItemValidation`, observation is centralized in `observeRecurring`, and there are no hosting-bridge quirks.

**Alternatives:** SwiftUI `NavigationSplitView` with `.toolbar` modifiers — simpler but encountered persistent toolbar layout issues. A hybrid AppKit-shell / SwiftUI-content app (the prior architecture) — workable, but the bridge shims (sheet/popover anchors, `WindowAccessor`) added complexity that pure AppKit removed.

### 2. VM bundle as `.kernova` package directory

**What:** Each VM is a directory with a `.kernova` extension containing `config.json`, the disk image, auxiliary storage, save files, and serial logs.

**Why:** Treats each VM as an atomic unit in Finder. Users can move, copy, or delete VM bundles as single items. The directory structure is predictable via `VMBundleLayout`, and `config.json` makes the format human-inspectable.

**Alternatives:** SQLite database, single directory with UUID-named files, or Core Data. The bundle approach is simpler and more transparent.

### 3. ASIF disk images via bundled templates

**What:** Disk images use Apple Sparse Image Format (ASIF), a macOS 26 format. Pre-built templates are stored lzfse-compressed in `DiskTemplates/` (~3 KB each) and decompressed at VM creation time.

**Why:** ASIF provides near-native SSD performance with space efficiency — a 100 GB disk image starts at ~4 MB on disk and grows as the guest writes data. The template approach is fully sandbox-safe (no subprocess spawning), unlike the previous `hdiutil`-based approach.

**Alternatives:** Raw disk images (simple but waste space), QCOW2 (not natively supported by Virtualization.framework), or runtime `hdiutil` invocation (not sandbox-safe, requires process spawning).

### 4. `@MainActor` isolation strategy

**What:** Everything touching `VZVirtualMachine` is `@MainActor`-isolated. `VMInstance`, `VirtualizationService`, and `MacOSInstallService` are all `@MainActor`. Services that don't touch VZ are `Sendable` structs with no mutable state.

**Why:** `VZVirtualMachine` is main-thread-only (Apple requirement). Swift 6 strict concurrency enforces this at compile time. Making the boundary explicit prevents accidental cross-thread access. `Sendable` structs for stateless services avoid unnecessary main-thread bottlenecks.

**Bridging:** Some `VZVirtualMachine` delegate callbacks arrive on the main thread but aren't marked `@MainActor` in the API. These use `nonisolated(unsafe)` with `MainActor.assumeIsolated` to bridge back.

**Alternatives:** Wrapping all VZ access in `MainActor.run {}` calls — more boilerplate and easier to miss a call site.

### 5. Service protocol abstraction

**What:** Every service has a corresponding protocol (`VirtualizationProviding`, `VMStorageProviding`, etc.) defined in `Services/Protocols/`.

**Why:** Enables dependency injection for testing. Mock implementations can track call counts, return canned responses, and inject errors via per-method `<method>Error` properties (e.g. `startError`). The coordinator and view models accept protocol types, not concrete implementations.

**Alternatives:** No protocols, test against real services — would require actual VM operations in tests, making them slow and environment-dependent.

### 6. VMLifecycleCoordinator separation

**What:** `VMLifecycleCoordinator` sits between `VMLibraryViewModel` and the services, orchestrating multi-step operations. It also enforces per-VM operation serialization — a token-based `[UUID: UUID]` dictionary maps each VM to its current operation token and rejects concurrent requests with `VMLifecycleCoordinator.LifecycleError.operationInProgress`. `stop`/`forceStop` bypass serialization entirely — they clear the token *before* calling the service, which invalidates any in-flight operation's `defer` guard and prevents stale removals.

**Why:** macOS VM installation is a multi-step process (download IPSW → create platform files → configure VM → install). Putting this in `VMLibraryViewModel` would bloat it with orchestration logic. The coordinator keeps the view model focused on list management and selection, while the coordinator handles operational complexity. Operation serialization prevents undefined behavior from concurrent `VZVirtualMachine` calls (e.g., double-start or pause-during-start).

**Alternatives:** Fat view model (simpler structure but harder to test and maintain), or individual operation objects (more granular but more types to manage).

### 7. Native NSToolbar with observation-driven validation

**What:** The main window and display window use `NSToolbar` with `NSToolbarDelegate` creating native `NSToolbarItem`s. Shared toolbar items (lifecycle, suspend, clipboard, pop-out, fullscreen, settings toggle) are managed by `VMToolbarManager`, a `@MainActor` `NSObject` subclass that handles item creation, state updates, and action routing for both controllers. Each controller configures it with an `instanceProvider` closure and a `Configuration` struct that captures per-controller differences (identifier strings, `isPreparing` checks, display capability gating, presence of the settings toggle). Toolbar state is driven by the shared `observeRecurring` helper (see Utilities), directly setting `isEnabled` on items on change. All shared items use `autovalidates = false`, because `NSToolbarItemValidation` returns `true` for any shared identifier when a VM is selected, and AppKit's autovalidation would otherwise fight the observation-driven `isEnabled` writes and produce a visible flicker on selection changes.

Item construction follows the glass-toolbar model documented in [docs/TOOLBAR.md](TOOLBAR.md): the lifecycle triplet is an `NSToolbarItemGroup` (a segmented control in its own capsule platter); Suspend, Pop Out, Fullscreen, and the settings toggle are plain bordered image items (`makeBorderedItem`); and the **clipboard item** is view-backed by `ClipboardToolbarButton` — a `.toolbar`-bezel `NSButton` pinned to the platter's 36×36 metric, hosting the Safari-style transfer bar as a real subview (a 22×6 capsule, centered, 3 pt above the circle's rim; opaque dynamic-gray track plus accent fill, drawn in `draw(_:)` so the colors resolve per-appearance). `VMToolbarManager.defaultItemIdentifiers` orders the default layout with fixed spaces choosing the glass capsule clusters: Suspend and Clipboard each in their own circle, the display pair together, the settings toggle on its own. `updateClipboardItem` sets the item's enablement from `canShowClipboard` and re-arms a manager-owned `observeRecurring` loop onto the selected VM's `transferProgress`, pushing the fraction (or idle `nil`) into the button's bar via `refreshClipboardTransferBar`; the dedicated loop keeps chunk-cadence redraws off the full toolbar-update path (#354). When an item swaps its image/label based on state (Start↔Resume, Pop Out↔Pop In, Fullscreen↔Exit Fullscreen, Show↔Hide Settings), the mutation is guarded behind a label-equality check so no-op updates don't trigger an AppKit redraw.

Both toolbars are user-customizable with autosaved configurations (#319). The manager's state-update lookups treat an absent item as a legitimate silent skip (the user may have removed it via customization) and warn only for a present-but-malformed lifecycle group. Because palette-added items are created in factory-default state and `autovalidates` is off, both controllers implement `toolbarWillAddItem` and refresh item state one runloop turn later. The main toolbar's identifier is `KernovaMainToolbar`; every VM's display window shares the single `KernovaVMDisplayToolbar` identifier so AppKit synchronizes customization across display windows and persists one configuration (window *frames* remain autosaved per-VM). The display window auto-hides its toolbar in fullscreen via the `.autoHideToolbar` presentation option (`window(_:willUseFullScreenPresentationOptions:)`) rather than toggling `toolbar.isVisible`, which would contaminate the autosaved configuration.

**Why:** Native `NSToolbarItem`s provide reliable layout, proper `.sidebarTrackingSeparator` support, and standard macOS toolbar appearance. The `observeRecurring` helper handles re-registration after each change and `[weak self]` teardown uniformly across every window controller, so toolbar updates stay reactive without SwiftUI and without duplicating the observation-loop boilerplate at each site. The shared `VMToolbarManager` eliminates ~150 lines of duplicated toolbar logic between `MainWindowController` and `VMDisplayWindowController`, ensuring toolbar changes are applied in one place.

**Alternatives:** SwiftUI `.toolbar` modifiers on a hosting controller — simpler declarative API but caused persistent layout issues with grouped items and sidebar tracking.

### 8. Apple Silicon-only (arm64)

**What:** The project sets `ARCHS = arm64` at the project level — no Intel (x86_64) slice is built, and the codebase contains no `#if arch(arm64)` conditionals.

**Why:** Virtualization.framework gates the macOS-guest APIs (`VZMacOSInstaller`, `VZMacOSRestoreImage`, `VZMacAuxiliaryStorage`, recovery start options) and VM save/restore (`saveMachineStateTo`/`restoreMachineStateFrom`) to Apple Silicon, so an Intel build would lose macOS guests *and* suspend/restore — and with a macOS 26 deployment target (the final Intel macOS release), the addressable Intel hardware is a handful of models on their last OS. The guards previously kept for a hypothetical Intel slice were never compiled (all dev and CI is arm64 Debug with `ONLY_ACTIVE_ARCH`), and the slice had silently stopped building.

**Alternatives:** Keep standard architectures with `#if arch(arm64)` guards and ship a Linux-guests-only Intel app — rejected as untested, already broken, and not worth ~38 conditional-compilation sites.

### 9. App Sandbox with per-path security bookmarks (#89)

**What:** The host app runs under the App Sandbox in every build configuration. Every user-picked external path (external `StorageDisk`s, `RemovableMediaItem`s, `SharedDirectory`s, Linux `kernelPath`/`initrdPath`, a local install IPSW) persists an app-scoped security bookmark in an optional `bookmark: Data?` field next to its raw path string in `config.json` (`SecurityScopedBookmark.capture` at the panel pick site). At each boot attempt, `VMInstance.openRuntimeFileAccess()` resolves every bookmark into a RAII `ScopedAccess`, heals stale bookmarks and moved paths back into the config (Apple's documented re-create pattern, persisted through `performConfigurationMutation`), and hands the scopes to `RuntimeFileAccess`; `tearDownSession()` — the chokepoint every stop/error/save/retry path funnels through — releases them exactly once. Removable-media scopes (cold-boot or hot-attach) are keyed by item id so a live eject releases exactly its own grant; existence probes (`AttachmentFileMonitor`, the delete sheet) and external-file trashing open momentary scopes via `SecurityScopedBookmark.fileExists` / a local `ScopedAccess`. The VM library needs none of this: `.applicationSupportDirectory` resolves into the container transparently.

**Why:** Mac App Store distribution requires the sandbox, and an open/save-panel grant dies with the process — bookmarks are the only sanctioned way to persist access across launches. Scopes are held for the whole VM runtime because VZ opens its fds at config-build time with no public retention guarantee, and unbalanced stop calls leak kernel resources until relaunch — so ownership sits in one place with one release point. A parallel optional field (rather than a path+bookmark wrapper type) keeps old `config.json` files decoding unchanged — a `nil` bookmark falls through to the raw-path attempt, which the sandbox denies for out-of-container files, surfacing the *existing* missing-file UX; re-picking the file mints a fresh bookmark. There is deliberately **no migration machinery** (no `Container Migration.plist`, no decode-time path rewriting): existing users drag their `.kernova` bundles onto the app once (the standard import flow — the drag grant is what lets the sandboxed app read the source) and re-pick any external attachments.

**Alternatives:** A `ScopedPath` wrapper struct (rejected: changes the JSON shape, forcing dual-format decoding — exactly the legacy machinery this project forbids); closing scopes right after `VZVirtualMachine.start()` (rejected: no API contract says VZ has finished opening; conservative whole-runtime scopes are trivially balanced); a `Container Migration.plist` (rejected: one-shot OS-level migration that moves user data invisibly, for a one-time step the import flow already handles transparently).

(`.accessory` apps are also App-Nap candidates, so the agent holds an `NSProcessInfo` activity while clipboard sharing is enabled to keep the 0.5 s poll from being throttled.) Each rep is pulled at most once and cached, so an image promised as both its UTI and `public.file-url` is fetched a single time. On the receive side, a small inline representation reassembles in RAM while a large inline one (past `maxResidentInlineBytes`) or a file representation streams to a local temp file under a free-space guard (`ClipboardFileStaging.makeSink`), a spilled inline rep being mmapped back so inline content has no size cap; a too-large file fails the pull's free-space pre-flight and the provider returns empty without a request, and a mid-flight or pre-Begin abort/teardown/timeout wakes the blocked pull and returns empty (the host Aborts every request it drops, so a supersession-mid-paste race wakes the pull immediately rather than parking to the 120 s backstop — #357). A non-image file rep resolves its `public.file-url` to the staged temp file; an image file rep is served inline for the image UTI and *also* as a staged file-url (so Notes shows the image and Finder can paste it) — the wire only ever carries the filename, never a host path. A superseding offer, a host `ClipboardRelease`, or a connection teardown retracts the promise and `failAll()`s the coordinator so any blocked provider returns immediately (the eager `InboundCollection`/`onTransferComplete`/`finishInbound`/`applyPairs`/`inboundStagingQueue` assembly is retired; the channel-wide receiver `onComplete`/`onAbort` is now a log-only fallback). Copied *files* (outbound; Finder ⌘C puts one `public.file-url` + name per item, not the bytes) are offered from a `stat` only — one disk-backed `.file` representation per file (read via `itemFileURLs`, each skipped individually if it's already inside the agent's own staging root), never read at offer time, no size cap, streamed 64 KiB at a time on request — the guest-side mirror of the host window's file-drop expansion. A copied **folder** is detected via `.isDirectoryKey` and archived **eagerly off the main queue** (`archiveAndOffer` hops to a global queue and back; an `archiveInFlight` flag stops overlapping 0.5 s polls double-archiving, and the completion drops itself if the clipboard changed mid-archive) into one `isDirectory` `.aar` rep staged in a dedicated `agent-send` `ClipboardFileStaging` (kept separate so an outbound archive's generation can't collide with an inbound transfer's). Inbound, a directory offer promises only `.fileURL`; `provideData` extracts the streamed `.aar` (re-tagged `isDirectory` from the offer in `pullRepresentation`) into a real folder under the staging root (`reserveDirectory`, double-disk free-space-guarded), cached per rep so a repeated pull reuses the extracted tree. **With `clipboard.dirtree.v1` negotiated (folder D1b, #422) this eager-archive path is only the fallback** (`peerSupportsDirTree` reads the control agent's Hello capability): a copied folder crosses as a `.directory` source rep with a stat-walk estimate (no archive), and inbound a directory rep — when the File Provider toggle is on — routes as a placeholder *tree* (`ClipboardDirectoryTree`: the guest fetches a `ClipboardTreeFetch` listing, builds a `FileProviderPublishFolder`, publishes the hierarchical manifest, and serves each child `fetchContents` via a child-addressed pull). The archive fallback (old peer / toggle off) archives at **request** time (`archiveAndStream`) rather than eagerly at copy. Echo suppression is digest-based (`ClipboardContent.digest`, a `.file` rep folding in its streamed sha256), reset on reconnect so a fresh host instance gets re-offered the current clipboard. The agent depends on the local `KernovaKit` SPM package for the wire types and channel implementation. Packaged into a disk image at build time by the "Package Guest Agent DMG" Run Script build phase, which `ditto`-copies the built `Kernova Guest Agent.app` (preserving its code signature) into the image. The disk image (containing the `.app` bundle, `install.command`, `uninstall.command`, and a LaunchAgent plist) is placed in `Contents/Resources/KernovaMacOSAgent.dmg`. Because the agent is embedded in that DMG resource at *build* time — where Xcode's export-time Developer ID re-signing can't reach — its Release config signs **Manual** with a `Developer ID Application` identity + a manually-created `Kernova macOS Agent Developer ID` provisioning profile (carrying the `group.app.kernova` App Groups authorization; not device-locked, so it validates inside an unregistered guest VM), `ENABLE_HARDENED_RUNTIME = YES`, and `OTHER_CODE_SIGN_FLAGS = --timestamp`, so the copied `.app` already carries its final notarizable distribution signature (Developer ID + hardened runtime + timestamp) when the DMG is baked. `SKIP_INSTALL = YES` keeps a `Product → Archive` a single-product `Kernova.app` archive (a multi-product 'generic' archive disables Distribute App). Hardened runtime is enabled on all five product targets (app, agent, both File Provider appexes, relaunch helper) in **Release only** — notarization requires it on every binary, while Debug and CI's ad-hoc / `CODE_SIGNING_ALLOWED=NO` builds are untouched. The full release + notarization flow lives in [RELEASING.md](RELEASING.md). At runtime, a single Virtual Machine-menu item (`toggleGuestAgentDisk`) manages this disk for the active VM. While the disk is **not** attached the item attaches it (mounting it as USB mass storage) with a purpose-framed title that hints at *why*, driven by `agentStatus`: Install (`.waiting`) / Update (`.outdated`) / Reinstall (`.expectedMissing`) / Manage (`.current` or `.unresponsive` — the agent is present but the user may want to reinstall or run `uninstall.command`). `.unresponsive` is treated like `.current` because it persists when the user disables/uninstalls/kills the agent in the guest; only the genuinely-transient `.connecting` leaves the item disabled. While the disk **is** attached the same item switches to **Eject Guest Agent Media** (a detach with no file deletion). All states require a live VM (`canAttachUSBDevices`). The mount purpose drives the post-attach instructions alert (`presentInstallerMounted(vmName:purpose:)`): `.install` points at `install.command`, `.manage` names both `install.command` and `uninstall.command`. The guest user runs `install.command`, which stages a `ditto` copy of the `.app` next to `~/Applications` (strips quarantine, validates the copy's version + code signature before touching an existing install), stops the old agent and File Provider extension with confirmed teardown, rename-swaps the validated bundle into `~/Applications` (restoring the previous bundle if the swap fails), cleans up any legacy bare-binary install, and registers a user-space LaunchAgent that execs the bundle's inner binary (`-y`/`--yes` runs either script non-interactively for test rigs); once the agent handshakes a current version the host **auto-ejects** the disk (see `VMInstance.onAgentBecameCurrent`). Eject is intentionally a menu-bar / Settings affordance only — it is *not* added to the sidebar agent-status popover or clipboard window, which stay install-focused. The vsock reconnect loop uses a flat 5s retry interval; `SO_RCVTIMEO` / `SO_SNDTIMEO` are set to 30 s as a safety net against wedged read/write calls. The build number is injected via `INFOPLIST_PREPROCESS`: a pre-Sources build phase ("Set Build Number from Git") calls the shared `Tools/set-build-number.sh agent`, which writes `#define AGENT_BUILD_NUMBER` (the squash-aware commit count scoped to `KernovaGuestAgent/ KernovaMacOSAgent/` — both the pre- and post-rename directory paths, so the count stays monotonic across the rename) to a header in `DERIVED_FILE_DIR`, and the explicit `Info.plist` references that macro for `CFBundleVersion`. As an app bundle the preprocessed plist is the bundle's real `Contents/Info.plist` (the bare-tool `CREATE_INFOPLIST_SECTION_IN_BINARY` embedding is no longer used), and `Bundle.main` reads version/build from it. The bundle carries an app icon (`Assets.xcassets` with an `AppIcon` set sharing the host app's icon; `ASSETCATALOG_COMPILER_APPICON_NAME = AppIcon`) so it shows a proper icon in Finder and System Settings → Login Items — there is no Dock icon at runtime (`.accessory`). The product is named `Kernova Guest Agent` (`PRODUCT_NAME`), so the wrapper is `Kernova Guest Agent.app` and `CFBundleName` carries the spaces that surface in Finder and Login Items (neither honors `CFBundleDisplayName` for this — Finder shows the wrapper filename, BTM shows `CFBundleName`); the executable and Swift module stay `KernovaMacOSAgent` (pinned via `EXECUTABLE_NAME` / `PRODUCT_MODULE_NAME`) so the LaunchAgent `ProgramArguments` path, the DMG build-phase inputs, and `install.command`'s `EXEC_REL` keep a space-free leaf. Because the synchronized-folder asset catalog only compiles into a target that has a `Resources` build phase, the target gained one when it became an application (the bare-tool form had none).
- **KernovaMacOSAgentFileProvider** — A sandboxed `NSFileProviderReplicatedExtension` (`.appex`) embedded in the guest agent's `Contents/PlugIns/` (#376). It makes a host→guest **file** paste lazy: when the host offers a single non-directory file rep, the agent's `FileProviderDomainHost` (the shared KernovaKit domain host, constructed in `AgentAppDelegate` and reached by the clipboard agent through the `FileProviderPublishing` seam) registers the `kernova-clipboard` File Provider domain, writes the offer into a small manifest in the shared app-group container, and puts a **dataless domain placeholder URL** on the guest pasteboard (constructed from the cached user-visible root) instead of a synchronous `public.file-url` promise — so a Finder paste returns instantly and never trips the 60s pasteboard-promise deadline. The extension's enumerator reads that manifest; on read, `fetchContents` decodes `(generation, repIndex)` from the item identifier and **relays the byte pull to the agent over the canonical `NSFileProviderServicing` anonymous-XPC connection** (#460; the appex is sandboxed and can't open vsock — the agent owns the vsock connection and pulls through the existing `ClipboardStreamReceiver`/`LazyPullCoordinator` off-main, with no deadline). Inverted wiring: the agent is the XPC client and exports the relay (sending an `ownerDidConnect()` handshake so the extension's listener accepts the connection); the extension calls it back at `fetchContents` — **asynchronously**, so it never blocks the framework's serialised reconnect — and a Darwin notification is the reconnect doorbell. The relay carries only the addressing and replies with a *path* — the agent stages the pulled bytes into the shared app-group container, and the extension APFS-clones that file into the domain's `temporaryDirectoryURL()` (same-volume, near-free) before handing it to the system. This **XPC relay is the codebase's first live-IPC surface** (a #145 trust-boundary down-payment: validate the XPC peer when hardened). The host is gated on host clipboard-sharing policy via the control agent's `onPolicy` seam, so nothing connects in a context that didn't enable clipboard (notably the CI test host). First-run enablement: macOS keeps a third-party File Provider extension **disabled** until the user enables it under System Settings → General → Login Items & Extensions → File Providers; the shared domain host detects the disabled state event-driven, off `NSFileProviderDomain.userEnabled` on the live system copy of the domain — an `NSFileProviderDomainDidChange` observer is the primary signal (the system posts it the instant the user flips the toggle), backstopped by a usage-triggered re-check in `publishSingleFile` and by `signalEnumerator` completion errors (a `signalEnumerator` probe alone is unreliable as the *initial* signal — its completion reports success even when the domain is disabled; the `-2011` that surfaces only on a later content fetch now triggers a re-probe instead of being discarded) — and the status item surfaces an "Enable in System Settings…" affordance (deep-linked via the shared `ClipboardFileProviderSettings.enablementDeepLinks`). The same availability observer also drives `VsockGuestClipboardAgent.fileProviderAvailabilityChanged` (#429): an offer that arrived before the domain settled — or before the user flipped the toggle — is written on the synchronous provider path, and without this, the live promise would stay there until the host re-copies; on the `.ready` transition the agent re-checks the live `inboundPromise` and, if it's still the eligible single-file case on the sync path and the guest clipboard hasn't changed since, re-publishes it through the File Provider (rewriting the pasteboard promise, preserving echo suppression via `lastPasteboardChangeCount`). Signed with the per-config app group (`KERNOVA_APP_GROUP`; #463): Debug uses the team-prefixed `$(DEVELOPMENT_TEAM).app.kernova`, which needs no provisioning profile and so never triggers the "access data from other apps" consent prompt inside an unregistered guest VM; Release uses the canonical `group.app.kernova` authorized by an embedded, manually-created Developer ID profile (**Manual** signing with a `Developer ID Application` identity + hardened runtime + `--timestamp`, because it is baked into the agent's DMG at build time where export-time re-signing can't reach — see KernovaMacOSAgent above; a Developer ID profile isn't device-locked, so it validates in an unregistered guest VM, #479). The host-side "Copy to Mac" File Provider mirrors the same group (#424). Scoped to **single file** in D1a; multi-file + folder reps and an inline fast-path follow in D1b/D1c. (The domain-host machinery — the agent's since-deleted `FileProviderHost` wrapper → the shared `FileProviderDomainHost`, the extension principal class, and `KernovaLogger`/the config/manifest/container helpers — was consolidated into `KernovaKit` in #424 Phase 1, parameterized by a `FileProviderConfig` (`.guest()`/`.host()`) so the guest and the host-side "Copy to Mac" extension share one implementation; #460 moved both onto the shared `FileProviderServicingConnector`, replacing the guest's direct Mach listener.)

- **KernovaFileProvider** — The host-side mirror that makes a guest→host **file** "Copy to Mac" lazy (#424). `KernovaFileProvider` is a sandboxed `NSFileProviderReplicatedExtension` (`.appex`) embedded in the **main app**'s `Contents/PlugIns/`, running the shared `FileProviderExtension` with `.host` config (domain `kernova-clipboard-host`, the same registered app group, a host-specific container subpath). The lazy flow mirrors the guest: `HostClipboardFileProvider.shared` (an app-level singleton — the Mac has one global pasteboard + one FP manifest, so the domain is owned once, not per-VM) registers the domain, publishes a single file rep as a dataless placeholder, and on read the extension's `fetchContents` relays the byte pull to the **main app** (the only process that can open vsock) over the canonical `NSFileProviderServicing` anonymous-XPC connection (#460) — the app is an ordinary resident app now, not a launchd Mach vendor. Inverted wiring: the app is the XPC client and, through `HostClipboardFileProvider`'s domain host + `FileProviderServicingConnector`, connects via `NSFileProviderManager.getService(named:for:)` on `.rootContainer` → `getFileProviderConnection` (identifier-based — no filesystem touch of the domain root, which the sandboxed app cannot read; #539), **exports** the relay (`FileProviderRelayService`, forwarding to the current VM's `pullStagedFile`, or `noSuchItem`/`serverUnreachable`), and sends an `ownerDidConnect()` handshake so the extension's listener accepts the connection (it fires `shouldAcceptNewConnection` only on the client's first message); the extension then calls that relay **back** at `fetchContents`, which relays the pull **asynchronously** so it never blocks the framework's serialised reconnect. The app warms the connection proactively — at publish and at domain registration — and reconnects on a Darwin reconnect doorbell / XPC invalidation; there is no Mach service, no broker, and no `summon` verb (a `.kernova` document-open reaches the single running instance directly). No bytes cross XPC — only `(generation, repIndex)` and a staged app-group path. Peer validation via `NSXPCConnection.setCodeSigningRequirement` (built by `FileProviderConfig.host()`, #476): the extension pins the app in `shouldAcceptNewConnection` (`ownerCodeSigningRequirement`, `identifier "app.kernova"` + team OU + `anchor apple generic`), and the app pins the extension on its control connection (`extensionCodeSigningRequirement`, same shape for `app.kernova.fileprovider`). The team OU is resolved at runtime from the running executable's own code signature (`KernovaCodeSignature.teamIdentifier()`, via `SecCodeCopySigningInformation`) rather than hardcoded, so the pin is automatically correct for whichever team built the app — and matches both Debug (Apple Development) and Release (Developer ID) signing from that team, since `anchor apple generic and certificate leaf[subject.OU]` doesn't distinguish between the two. The byte pull funnels through `VsockClipboardService.pullStagedFile` — a synchronous-blocking bridge (shared `LazyPullCoordinator`, thread-aware) that serves **both** the relay (off-main XPC queue) and the toggle-off paste fallback (on the main-thread `provide` callback). With the File-Providers toggle off (the out-of-box default), "Copy to Mac" of a file falls back to a size-capped synchronous paste (`ClipboardStreamTuning.maxDeadlineSafeFileBytes`, 256 MiB — must finish within Finder's ~60 s deadline); larger files are dropped. The disabled state is detected via `NSFileProviderDomain.userEnabled` (event-driven in the shared domain host — an `NSFileProviderDomainDidChange` observer, backstopped by a usage-triggered refresh in `publishSingleFile` and `signalEnumerator` error feedback, rather than a poll) and surfaced in the clipboard window by a `ClipboardEnablementBanner` (`HostClipboardFileProvider.shared.availability == .needsEnabling`), deep-linking to System Settings. The eager `ClipboardFileStaging.adopt()` bridge is removed. Signed with the shared per-config app group (`KERNOVA_APP_GROUP`; #463) like the guest appex — Debug team-prefixed, Release `group.app.kernova` + an embedded provisioning profile. Unlike the guest appex, the host app and this embedded appex sign `Automatic` with `REGISTER_APP_GROUPS = YES` (so Xcode 16.3+ automatic signing generates a Developer ID profile carrying the group) + hardened runtime, and are re-signed with Developer ID at Organizer export — no build-time Developer ID identity is needed because nothing embeds them in a DMG (#479). D2 is **single non-directory file**; directories stay eager (#422). Host-only — no guest agent version bump.

- **KernovaMacOSAgentTests** — Standalone unit-test bundle (no `TEST_HOST` / `BUNDLE_LOADER`) that covers the agent-side classes. Because `KernovaMacOSAgent` is an application target (not a framework), its symbols are not linkable — the test bundle instead compiles the agent's Swift source files directly (all except `AgentAppDelegate.swift` — the `@main` entry — plus `Info.plist`, `Assets.xcassets`, and the shell scripts, excluded via `PBXFileSystemSynchronizedBuildFileExceptionSet`). This direct compilation makes all `internal` members accessible without `@testable import`, which is unavailable for tool targets. Three agent classes required light testability seams: `VsockGuestClient` gained a `socketProvider` closure injection point and a parameterized `retryInterval` (default unchanged at 5 s); `VsockGuestClipboardAgent` gained a `Pasteboard` protocol (with `NSPasteboard` conformance) and injected `client`/`pasteboard` init parameters; `VsockHostConnection` lifted `pendingLogs`, `lock`, `bufferFrame`, `flushPendingLogs`, and `logBufferLimit` from `private` to internal. Bundle-specific test helpers live in `TestHelpers.swift` (socket-pair factories, `nextFrame`, `awaitFirst`, `AtomicInt`, frame factories); the wait primitives (`AsyncGate`, `waitUntil`, `TestFailure`) come from the shared `KernovaTestSupport` product (#526 — see TESTING.md's async-wait table). `KernovaLogMessageTests` moved to `KernovaKitTests` along with the `KernovaLogger`/`KernovaLogMessage` sources (#434). Non-parallelizable in the scheme because each test worker loads the agent sources which include global state (`VsockLogBridge.connection`); tests share one runner process.

### Mocked but Not Directly Tested

These services interact with system processes, the network, or VZ installer internals. They are fully mocked in other tests but don't have their own test suites against real implementations:

- `DiskImageService` — decompresses bundled templates (no subprocess; direct testing feasible but requires bundled resources in test target)
- `IPSWService.fetchLatestRestoreImageURL()` — a thin wrapper over `VZMacOSRestoreImage.latestSupported` that hits Apple's network (the download/resume flow itself is now directly tested — see `IPSWServiceDownloadTests` in the Well Covered table)
- `MacOSInstallService.install(into:restoreImageURL:progressHandler:)` — requires a real `VZVirtualMachine` and restore image (the restore-image path resolution that precedes both VZ hand-offs *is* directly tested — see `MacOSInstallServiceTests` in the Well Covered table)

### Not Tested

- `SecurityScopedBookmark` / `ScopedAccess` / `RuntimeFileAccess` runtime behavior — bookmark creation/resolution needs a signed, sandboxed process holding a real panel grant, which the unit-test environment can't fabricate (CI runs unsigned, so the sandbox isn't even enforced there); covered by the manual sandbox verification pass. The model-level persistence (bookmark fields round-tripping `config.json`, clone carry-through, nil-decode for pre-sandbox configs) *is* unit-tested in `VMConfigurationTests` / `VMConfigurationCloneTests` / `MacOSInstallContextTests`
- `VMDirectoryWatcher` — relies on `DispatchSource` file system monitoring
- `SystemSleepWatcher` — relies on `NSWorkspace` sleep/wake notifications (sleep/wake logic tested via `VMLibraryViewModel`)
- `KernovaUTType` — static UTType declaration
- All window controllers (`MainWindowController`, `VMDisplayWindowController`, `ClipboardWindowController`)
- `AppDelegate` — window management and the live app-lifecycle paths; the extracted quit-classification logic (`classifyQuit`) is tested via `AppDelegateQuitClassificationTests`, the honest app-menu quit model (`appMenuQuitItems`) via `AppDelegateQuitMenuTests` (#624), the cold-launch decisions (`launchProvenance`, `coldLaunchOutcome`) via `AppDelegateColdLaunchTests`, the untracked-panel classifier (`isUntrackedUserPanel`) via `AppDelegateUntrackedPanelTests`, and the status-item open routing (`statusItemOpenTarget`) via `AppDelegateStatusItemOpenTests`
- Pure AppKit view rendering beyond the extracted testable helpers (`DetailRoute`, `MicPermissionPresentation`) and the VC-level tests (`VMSettingsViewControllerTests`, wizard-step tests)

### Test Patterns

- **Framework:** Swift Testing (`@Suite`, `@Test`, `#expect`) — not XCTest
- **Mocks:** 9 service-protocol mocks supporting call counting and error injection via per-method `<method>Error` optionals such as `startError` / `createVMBundleError` (plus `MockVMLibraryPresenting`, a presentation-delegate recorder without error injection). Includes `SuspendingMockVirtualizationService` for testing operation serialization and `SuspendingMockUSBDeviceService` for testing the mount mutex in `mountGuestAgentInstaller` — both suspend mid-operation to verify concurrent rejection. Rely on `@MainActor` cooperative scheduling (documented in the mocks) and enforce single-suspension via `precondition`
- **Factories:** Shared helpers like `makeInstance()`, `makeViewModel()`, `makeCoordinator()` reduce setup duplication
- **Error paths:** Mocks support setting a per-method `<method>Error` optional (e.g. `startError`) to inject failures and verify error handling
