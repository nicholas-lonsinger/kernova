# Architecture

## Overview

Kernova is a macOS application for creating and managing virtual machines via Apple's Virtualization.framework, supporting both macOS and Linux guests. It is built as a pure-AppKit app (the app target no longer imports SwiftUI), targeting macOS 26 (Tahoe) with Swift 6 strict concurrency. There are no external package dependencies ‚Äî the app uses only Apple system frameworks.

> The clipboard subsystem (host‚Üîguest copy/paste) is governed by a dedicated set of design principles in [CLIPBOARD.md](CLIPBOARD.md) ‚Äî the authoritative *why* and rules behind the clipboard files mapped below. Consult it before changing any clipboard code.

### App Layer

**Files:** `AppDelegate.swift`, `MainWindowController.swift`, `SettingsWindowController.swift`, `DetailContainerViewController.swift`, `VMDisplayWindowController.swift`, `VMToolbarManager.swift`, `ClipboardWindowController.swift`

`AppDelegate` is the entry point. It creates the `VMLibraryViewModel` and `VMLifecycleCoordinator`, opens the main window, and manages the lifecycle of all windows. It tracks window controllers in dictionaries keyed by VM UUID, enabling one-to-many relationships (a VM can have a main view, a fullscreen window, and a clipboard window open simultaneously). `AppDelegate` also handles:
- The application menu (including VM-specific actions, Force Stop with `canForceStop` validation, and Window > Show Library with Cmd+0). The View menu hosts the toolbar commands ‚Äî "Show/Hide Toolbar" (‚å•‚åòT, `NSWindow.toggleToolbarShown(_:)`) and "Customize Toolbar‚Ä¶" (`NSWindow.runToolbarCustomizationPalette(_:)`) ‚Äî as nil-target standard actions that resolve against the key window, so AppKit retitles/validates them without any `validateMenuItem` involvement; AppKit augments the View menu with its automatic window-tabbing items, while fullscreen/tiling commands land in the Window menu's AppKit-populated section (#318, #319)
- The Settings window (`Settings‚Ä¶` / ‚åò,): `showSettings(_:)` lazily creates and retains a `SettingsWindowController` singleton
- Advanced lifecycle actions ‚Äî "Start in Recovery Mode" and "Force Stop" ‚Äî follow a uniform surfacing rule: **always visible in the Virtual Machine menu bar** (the full discoverability surface, where `validateMenuItem(_:)` greys out what doesn't apply), and **Option-revealed in the sidebar context menu** (gated by `AppPreferences.alwaysShowAdvancedOptions`, which shows them plainly when on). The preference governs only the sidebar reveal, not the menu bar.
- "Start in Recovery Mode" ‚Äî gated on `VMInstance.canStartInRecovery` (stopped macOS guests). Menu bar: an always-visible item at ‚å•‚åòR. Sidebar: the Option-alternate of "Start". Selecting it routes through `VMLibraryViewModel.confirmStartInRecovery` ‚Üí a `DetailAlertsPresenter` confirmation ‚Üí `startInRecoveryConfirmed`
- "Rename" (no ellipsis ‚Äî it starts an inline edit like Finder's single-item Rename, not a dialog) ‚Äî reveals the sidebar surface, then routes to its inline rename: the handler fronts the library window (`showLibraryWindow`), uncollapses the sidebar (`MainWindowController.revealSidebar`), and calls `VMLibraryViewModel.renameVMInSidebar`. The reveal step is load-bearing ‚Äî the settings pane‚Äôs Name field sits hidden behind a running VM‚Äôs live display (the original #320 no-op), and the sidebar itself can be collapsed or in a non-key window, so no surface is *unconditionally* visible; the command makes its target visible rather than assuming it. The settings Name field remains the click-to-rename affordance for the detail surface itself
- Lifecycle labels are status-dependent and shared across all three surfaces (menu bar, sidebar context menu, toolbar) via `VMInstance` display helpers: `startAction` ("Start" / "Install" / "Resume Install" per pending-install state) and `stopActionMenuTitle` / `stopActionToolbarLabel` ("Stop", or "Discard Saved State‚Ä¶" for a cold-paused VM, naming the destructive consequence ‚Äî #321). The menu bar retitles in `validateMenuItem(_:)`; the toolbar in `VMToolbarManager.updateLifecycleGroup`
- "Show in Finder" ‚Äî reveals the VM bundle via `NSWorkspace.activateFileViewerSelecting`. Present in both the menu bar and the sidebar context menu, and (uniquely among VM menu-bar actions) stays enabled while the VM is preparing, matching the sidebar's preparing menu ‚Äî the bundle already exists on disk
- "Open VMs Folder" (File menu) ‚Äî opens the VMs storage folder (`VMStorageService.vmsDirectory`, which creates it when missing) as a Finder window via `NSWorkspace.open`. VM-independent and always enabled, including while a VM is preparing and with no VM selected. Titled "Open ‚Ä¶ Folder" (a window of the folder's contents, like the Script menu's "Open Scripts Folder") rather than "Show in Finder" (reveal-selected-in-parent), and "VMs Folder" rather than "Library", which the Window menu already uses for the main window (#333)
- "Force Stop" ‚Äî gated on `VMInstance.canForceStop` (`VMStatus.canForceStop` minus cold-paused, where there is no live VM to terminate and the retitled stop item ‚Äî "Discard Saved State‚Ä¶" ‚Äî is the single surface for that action). Menu bar: an always-visible sibling of "Stop". Sidebar: when a VM is gracefully stoppable (`VMInstance.canStop`) it is the Option-alternate of "Stop" (no top anchor needed ‚Äî a Pause/Resume item always precedes the Stop pair); for transient states where graceful stop is unavailable (`canForceStop && !canStop`: starting/saving/restoring) it appears as a standalone, always-visible item. "Delete Immediately‚Ä¶" is the Option-alternate of "Move to Trash‚Ä¶" in **both** surfaces: the sidebar context menu (gated by `alwaysShowAdvancedOptions` like the other advanced alternates) **and** the menu bar (‚åò‚å´ ‚Üí ‚å•‚åò‚å´). - Suspend-on-quit behavior (suspending VMs before termination)
- Conditional termination: `applicationShouldTerminateAfterLastWindowClosed` returns `false` when VMs are active or fullscreen windows exist, preventing premature exit on fullscreen-to-inline transitions
- Dock icon reopen: `applicationShouldHandleReopen` restores the main window when clicked with no visible windows, or when clicked while the app is already active and the library window has been closed

The helper monitors the app's PID via `DispatchSource` and relaunches the app via `NSWorkspace` after it exits, working around macOS's TCC relaunch timeout. TCC is positively identified by intercepting the `kAEQuitApplication` Apple Event and checking if the sender is the Privacy & Security settings extension (`com.apple.settings.PrivacySecurity.extension`).

`MainWindowController` creates an `NSWindow` with an `NSSplitViewController` as the content view controller. The split view has two panes: a sidebar (`NSSplitViewItem(sidebarWithViewController:)` wrapping the pure-AppKit `SidebarViewController` ‚Äî a source-list `NSOutlineView`) and a detail pane (`DetailContainerViewController`, which hosts the AppKit empty-state ‚áÜ `VMDetailRouterViewController`). The settings toggle only appears in the main window ‚Äî the pop-out display window passes `settingsToggleID: nil` ‚Äî while the clipboard item appears in both windows (its nil-target `showClipboard` action targets the key display window's VM via `AppDelegate.activeInstance`). The main window's customize palette additionally offers Clone, Show in Finder, and Move to Trash as palette-only items (not in the default set) ‚Äî VM-scoped verbs only; app-global commands like "Open VMs Folder" stay menu-bar-only ‚Äî whose enablement mirrors the menu bar's `validateMenuItem(_:)` predicates via `NSToolbarItemValidation`. ### Models

**Files:** `VMConfiguration.swift`, `VMInstance.swift`, `VMBundleLayout.swift`, `VMStatus.swift`, `VMBootMode.swift`, `VMGuestOS.swift`, `MacOSInstallState.swift`, `KernovaUTType.swift`

The model layer has two key types:

- **`VMConfiguration`** is the persisted identity of a VM. It's a `Codable` + `Sendable` struct written as `config.json` inside each VM bundle. It holds: name, UUID, guest OS type, boot mode, CPU/memory/disk settings, display configuration (including `lastFullscreenDisplayID` for remembering which display a VM was fullscreen on), network settings, audio settings (`audioInputEnabled` ‚Äî opt-in host microphone passthrough, defaults to off; `audioOutputEnabled` ‚Äî guest audio routed to the host's audio output, defaults to on), guest agent state (`agentLogForwardingEnabled` for the opt-in vsock log fan-in, and `lastSeenAgentVersion` ‚Äî the most recent guest-reported `Hello.agent_info.agent_version`, used to suppress the install nudge on stopped VMs whose agent has previously connected and to arm the post-start "didn't reconnect" watchdog), and OS-specific fields (macOS hardware model data, Linux kernel/initrd/cmdline paths).

- **`VMInstance`** is the runtime representation. It's an `@Observable` `@MainActor` class that wraps a `VMConfiguration`, an optional `VZVirtualMachine`, and a `VMStatus`. It references the VM's bundle path and provides computed properties for disk image, aux storage, and save file locations via `VMBundleLayout`. A view-layer extension (`VMInstance+Display.swift`) provides display properties (`statusDisplayName`, `statusDisplayColor`, `statusToolTip`) that distinguish preparing VMs (shown as "Cloning‚Ä¶"/"Importing‚Ä¶" in orange with a spinner), cold-paused VMs (state saved to disk, shown as "Suspended" in orange), and live-paused VMs (in memory, shown as "Paused" in yellow), plus the shared lifecycle action labels (`startAction`, `stopActionMenuTitle`, `stopActionToolbarLabel`) consumed by the menu bar, sidebar context menu, and toolbar so every surface titles the same action identically. The `PreparingOperation` enum (`.cloning`, `.importing`) provides display labels, cancel labels, and alert titles for preparing states. The `PreparingState` struct bundles the operation and its cancellable task into a single optional (`preparingState`) ‚Äî when non-nil the instance is preparing, and `isPreparing` is a computed convenience. A per-instance `detailPaneMode` (enum `DetailPaneMode { case display, settings }`, defaulting to `.display`) lets the user toggle, while the VM is running, between the live display and a read-only view of the settings form; the mode is ignored when the VM is stopped (settings are always shown then). For macOS guests, `VMInstance` also owns a one-shot post-start watchdog: `startAgentPostStartWatchdog(grace:)` (kicked from `VirtualizationService.start` once status reaches `.running`) waits for the grace period (default 120 s) and flips `agentExpectedButMissing = true` if no guest agent `Hello` arrived in that window; the flag is cleared by `recordObservedAgentVersion(_:)` (called via the `VsockControlService` `onAgentVersionObserved` callback) and by `tearDownSession`. `VMInstance.agentStatus` synthesizes the `.expectedMissing(expected:)` case from `agentExpectedButMissing` + `configuration.lastSeenAgentVersion` so the rest of the UI sees the case alongside live `VsockControlService`-sourced states. Configuration mutations driven by guest activity (e.g. persisting a new `lastSeenAgentVersion`) flow through the `onUpdateConfiguration` closure, which `VMLibraryViewModel` wires to its centralized `updateConfiguration(of:mutate:)` dispatcher at every instance-construction site ‚Äî so guest-driven and user-driven mutations share one persist + apply-live-policy path. `VMInstance` also exposes an `onAgentBecameCurrent` closure (wired the same way in `wirePersistence(for:)`) that `recordObservedAgentVersion(_:)` fires when a newly-observed agent version is current ‚Äî i.e. an install/update just completed ‚Äî driving the host-side auto-eject of the guest-agent installer disk via `VMLibraryViewModel.unmountGuestAgentInstaller`, so it runs regardless of which (if any) window is open. The runtime list of currently-attached USB mass storage devices (whether sourced from `removableMedia` at cold start or from later XHCI hot-attaches) lives on `VMInstance.liveRemovableMedia: [USBDeviceInfo]`, which the reconcile flow keeps in sync with the actual VZ device list.

`VMBundleLayout` is a `Sendable` struct that takes a bundle root path and provides all derived file paths (disk image, aux storage, save file, serial log, etc.), keeping path logic centralized.

The remaining models are enums: `VMStatus` (stopped/starting/running/paused/saving/restoring/installing/error), `VMBootMode` (macOS/efi/linuxKernel), `VMGuestOS` (macOS/linux), and `MacOSInstallState` (tracking download and install phases with progress). `VMStatus` provides computed properties for state checks (`canStart`, `canStop`, `canForceStop`, `canPause`, `canResume`, `canSave`, `canEditSettings`, `canRename`, `isTransitioning`, `isActive`). `canForceStop` covers all states where a `VZVirtualMachine` may exist and need forceful termination (running, paused, starting, saving, restoring); the instance-level `VMInstance.canForceStop` additionally excludes cold-paused (nothing in memory to terminate ‚Äî discarding the saved state is that state's equivalent action), mirroring the `canStop`/`canSave` instance-level refinements.

Services are split by concurrency requirements:

- **`@MainActor` services** (interact with `VZVirtualMachine`):
  - `VirtualizationService` ‚Äî start, stop, pause, resume, save state, restore state. `start(_:)` has two branches: restore from a save file, or cold-boot (build a fresh `VZVirtualMachineConfiguration` and attach a new VM). The cold-boot path retries with bounded escalating backoff (`fileLockRetryDelay`, ~3.75 s budget) when the start fails on VZ file-lock contention, classified by `isFileLockContention`: `.invalidVirtualMachineConfiguration` carrying POSIX `EAGAIN` underneath (the "Failed to lock auxiliary storage" error, or its disk-image sibling) ‚Äî matched by domain/code, never by localized text. A previous `VZVirtualMachine` on the same bundle releases its advisory file locks only at *deallocation*, which lags `vm.state == .stopped` in proportion to guest memory size, so the post-install auto-boot (and a rapid manual stop‚Üístart) can transiently collide with the dying VM ‚Äî see `MacOSInstallService` below for the state-gating half of this story. Lock contention is also a transient start error (`isTransientStartError`), so even an exhausted retry leaves the VM `.stopped` (grey, retryable) rather than `.error`.
  - `MacOSInstallService` ‚Äî loads restore image, creates platform files (aux storage, hardware model, machine identifier), runs `VZMacOSInstaller` with KVO progress tracking. After `installer.install()` resolves, it explicitly **waits for `vm.state` to reach `.stopped`** before returning. VZ's `install` completion handler fires while the post-install guest shutdown is still propagating through the framework's state machine ‚Äî without the wait, the caller's auto-boot would cold-rebuild a `VZMacAuxiliaryStorage(contentsOf:)` while the install-side instance still held the file lock, producing the "Failed to lock auxiliary storage" error. Waiting also gives our `VZVirtualMachineDelegate.guestDidStop` a chance to fire, which releases our refs via `resetToStopped`; if the delegate doesn't fire within the timeout the install service tears down explicitly as belt-and-braces. The state gate is necessary but not sufficient: the advisory lock is released at the install-time VM's *deallocation*, which lags `.stopped` (more guest RAM ‚Üí longer teardown ‚Äî a 32 GB guest reliably outlasts the gate on Apple Silicon), and VZ offers no API to observe the release ‚Äî `VirtualizationService`'s cold-boot retry (above) covers that residual window.

- **`Sendable` struct services** (no mutable state, safe to call from anywhere):
  - `VMStorageService` ‚Äî creates/deletes/lists VM bundle directories at `~/Library/Application Support/Kernova/VMs/` and handles cloning (deep copy with new UUID). Deletion has two dispositions: `deleteVMBundle` (move to Trash via `trashItem`) and `permanentlyDeleteVMBundle` (immediate `removeItem`, the user-confirmed bypass-Trash path)
  - `DiskImageService` ‚Äî creates ASIF disk images by decompressing bundled lzfse-compressed templates (sandbox-safe, no subprocess)
  - `IPSWService` (`final class` for `URLSession` lifecycle) ‚Äî fetches available macOS restore images from Apple's catalog and downloads IPSW files directly into a Finder-visible `.kernovadownload` bundle (`Info.plist` + `data` at the bundle root). Streams response chunks via a `URLSessionDataDelegate` bridged to an `AsyncThrowingStream<Data, Error>` (the per-byte overhead of `URLSession.bytes(for:)` is unacceptable at multi-GB scale); manual HTTP `Range` / `If-Range` against the IPSW CDN drives resume, with the data file's on-disk size as the resume offset. On completion the `data` file is moved to the user-chosen `.ipsw` destination and the bundle is trashed. Cancellation safety: `streamBytes` re-checks `Task.checkCancellation()` **after** the for-await loop, because `AsyncThrowingStream.next()` resolves to `nil` (not throw) when the consumer's task is cancelled while parked ‚Äî without that post-loop check, a user-cancelled download would proceed past the loop and finalize partial bytes onto the destination. A companion byte-count check (`totalWritten == expectedTotal`) catches the rarer "server closed cleanly under Content-Length" case. On any throw from this region the bundle is preserved so the next Start can resume from the partial bytes. Stale bundles are auto-discarded when the stored `originalURL` differs from the caller's request or when `Info.plist` fails to decode. The `.kernovadownload` UTI conforms to `com.apple.package` so Finder shows the bundle as a single icon

- **`SystemSleepWatcher`** ‚Äî `@MainActor` observer class that monitors `NSWorkspace.willSleepNotification` and `NSWorkspace.didWakeNotification`. Follows the same pattern as `VMDirectoryWatcher`: callback-driven, `nonisolated(unsafe)` for observer tokens, `start()`/`deinit` lifecycle. Owned by `VMLibraryViewModel`, which uses it to auto-pause running VMs before sleep and resume them on wake.

- **`ConfigurationBuilder`** ‚Äî Translates a `VMConfiguration` into a `VZVirtualMachineConfiguration`. Handles three boot paths: `VZMacOSBootLoader` (macOS), `VZEFIBootLoader` (EFI/UEFI), and `VZLinuxBootLoader` (direct kernel boot). Configures CPU, memory, storage, network, display, keyboard, trackpad, and audio devices. The virtio sound device carries an input stream when `audioInputEnabled` and an output stream when `audioOutputEnabled`; when both are off the sound device is omitted entirely. When `clipboardSharingEnabled` is set on a Linux guest, configures a `VZVirtioConsoleDeviceConfiguration` with a SPICE-named port using raw `VZFileHandleSerialPortAttachment` pipes (not `VZSpiceAgentPortAttachment`); macOS guests instead carry clipboard over the vsock device. For macOS guests it always appends a `VZVirtioSocketDeviceConfiguration` so the host can install vsock listeners (log + clipboard) against the live device once the VM is created. Resolves symlinks on user-supplied paths (shared directories, kernel/initrd, ISO images) and validates them before passing to VZ. File paths (kernel, initrd, ISO) are checked for existence and rejected if they point to directories. Shared directory validation checks existence, is-directory, readability, and writability (for read-write shares) against the resolved path.

- **`SpiceAgentProtocol`** ‚Äî Pure data types and parsing for the SPICE agent wire format. Defines VDI chunk headers, VDAgent message headers, clipboard message types, and capability bitmasks. Includes `SpiceMessageBuilder` (builds wire-ready messages with a `port` parameter defaulting to `serverPort` for host-side use) and `SpiceAgentParser` (incremental parser handling fragmented data across multiple pipe reads). Fully `Sendable`, no I/O. Used only by the host-side `SpiceClipboardService` (Linux clipboard transport) ‚Äî macOS guests now use vsock instead, so the file is no longer multi-target.

- **`AgentStatus`** ‚Äî Enum (`.waiting | .current(version) | .outdated(installed, bundled) | .unresponsive(version) | .expectedMissing(expected) | .connecting(expected)`) that drives install/update/unresponsive/reinstall affordances in the sidebar and clipboard window, plus the Virtual Machine-menu guest-agent item's purpose-framed title (Install/Update/Reinstall/Manage) and attach-vs-eject mode. Sourced from `VsockControlService` for macOS guests (independent of clipboard sharing) and from `SpiceClipboardService` for Linux guests (`.waiting` / `.current` only). The UI reads it via `VMInstance.agentStatus`, which dispatches by `configuration.guestOS` and synthesizes `.expectedMissing(expected:)` / `.connecting(expected:)` from the post-start watchdog flag plus `configuration.lastSeenAgentVersion` ‚Äî `VsockControlService` itself does not (and cannot) produce those cases because it has no access to persisted host state. The static `isObservedVersionCurrent(_:bundled:)` helper is the single source of truth for current-vs-outdated classification, shared by `VsockControlService.agentStatus` and the host-side auto-eject decision.

- **`ClipboardServicing`** ‚Äî `@MainActor` protocol covering the public surface (`clipboardContent` ‚Äî a `ClipboardContent` of ordered UTI-tagged representations, `isConnected`, `supportsBinaryRepresentations`, `lastTransferIssue`, `stop()`, `grabIfChanged()`, plus the lazy-receive hooks `materializeForPreview() async` and `materializeForCopy() async -> ClipboardContent`) shared by both clipboard implementations. The two materialize hooks have default no-op extension impls (return `clipboardContent` unchanged) so eager transports inherit them; only `VsockClipboardService` overrides them to pull lazily-offered guest representations on demand. `VMInstance.clipboardService` holds the existential, so the clipboard window controllers don't branch on transport. Agent install/version state is *not* part of this protocol ‚Äî for macOS guests it lives on `VsockControlService` (the always-on control channel, independent of whether clipboard sharing is enabled). `SpiceClipboardService` exposes its own `agentStatus` directly for Linux.

- **`KernovaGuestAgentInfo`** ‚Äî Static accessors for the bundled guest agent. The "Package Guest Agent DMG" build phase extracts `CFBundleShortVersionString` from the built `Kernova Guest Agent.app/Contents/Info.plist` (via `plutil -extract`) and writes it to `Resources/KernovaGuestAgentVersion.txt`; this file is the single source of truth for "what version did I bundle" and cannot drift from the actual agent binary. The accessor reads the sidecar lazily; missing/empty files trip an `assertionFailure` since the build phase is required to produce them. Also exposes the installer DMG URL.

- **`SpiceClipboardService`** ‚Äî `@MainActor` `ClipboardServicing` implementation for Linux guests. Reads from the guest pipe on a background GCD queue, parses messages via `SpiceAgentParser`, and exposes `clipboardContent` (observable, editable by the UI) and `isConnected`. Text-only transport: only the UTF-8 text representation crosses the wire, and `grabIfChanged()` is a no-op for content without one (`supportsBinaryRepresentations == false`, so the window's intake explains rather than silently accepting; rich SPICE formats are tracked as an issue #112 follow-up). When the clipboard window loses focus, `grabIfChanged()` sends a `CLIPBOARD_GRAB` if the text was edited. Uses raw pipes rather than `VZSpiceAgentPortAttachment` so clipboard data flows through the gated UI instead of the host `NSPasteboard`.

- **`VsockClipboardService`** ‚Äî `@MainActor` `ClipboardServicing` implementation for macOS guests, layered on `VsockChannel`, driving the chunk-streamed clipboard protocol via `ClipboardStreamSender` / `ClipboardStreamReceiver` (in `KernovaKit`). An outbound clipboard change is announced as a **metadata-only** `ClipboardOffer` carrying a `ClipboardRepresentationInfo` per representation (`uti`, `byteCount`, `filename`, `isInline`, `isDirectory`) under a monotonically increasing `generation` ‚Äî no bytes leave at offer time. The offer also carries an offer-level `is_concealed` flag (set when the source pasteboard marked the snapshot `org.nspasteboard.ConcealedType`); a concealed inbound offer publishes concealed content that the window hides behind a placeholder and `materializeForPreview()` skips (the secret is never pulled into host memory for a preview), though `materializeForCopy()` still streams it on an explicit Copy-to-Mac. **Inbound is a lazy hybrid.** `handleOffer` publishes a metadata-only `.pendingRemote` placeholder per offered rep to `clipboardContent` immediately (an `InboundPromise` holds the offered rep metadata, in offer order, plus a per-rep materialization cache) and pulls **nothing**; the clipboard window pulls only the reps it renders richly ‚Äî text, inline RTF, and images up to `ClipboardPreviewPolicy.maxEagerPreviewBytes` (32 MiB) ‚Äî **on display** via `materializeForPreview()` (idempotent per generation, started from `viewDidAppear`/status-update when the window is visible), and `materializeForCopy()` pulls every remaining rep when the user picks "Copy to Mac". Each rep is pulled at most once (cached in the promise). A single `pull(...)` bridges the off-actor stream receive to `async`/`await` via a `CheckedContinuation` resumed by the receiver's per-`transfer_id` `awaitTransfer` hook ‚Äî wrapped in a single-resume `PullContinuation` box, since the off-main awaiter, the on-main send-failure catch, and a `lazyPullTimeout` backstop all race a channel teardown (the first resume wins, the rest are no-ops; the backstop also resolves a pull the guest accepts but never streams while the channel stays open); it answers each request with a `ClipboardStreamBegin` ‚Üí `ClipboardChunk`* ‚Üí `ClipboardStreamEnd` sequence under windowed flow control (`ClipboardStreamAck` advances the credit window; `ClipboardStreamAbort` tears a transfer down), with a streaming SHA-256 verified at End. A small inline representation reassembles in RAM; a large inline one (past `maxResidentInlineBytes`) or a file representation streams to a temp file under the receiver's free-space guard, never resident whole ‚Äî a spilled inline rep is served back via a memory-mapped read so inline content has no size cap, and large files never cross the wire until the user copies them. `grabIfChanged()` refuses to offer content that still holds `.pendingRemote` placeholders back to the guest (the sender can't stream a placeholder and received content shouldn't echo). A **copied file** (outbound) is offered from a `stat` only ‚Äî a disk-backed `.file` representation that is never read at offer time and has no size cap ‚Äî and streamed on request. **Multiple files** ride as several `.file` reps in one offer (`ClipboardContent` partitions reps by filename: each non-empty filename is a distinct file, the rest are one inline item); the inbound side materializes each and the window writes one `NSPasteboardItem` per file (`hostPasteboardItems`), so a Finder paste creates N files. A **folder** (or OS package) rides as one `.file` rep flagged `isDirectory` whose bytes are an in-process AppleArchive (`ClipboardDirectoryArchive`, `.aar`/LZFSE) of the tree: the sender archives eagerly off the main actor into its staging (the offer needs the archive's size and the stream its SHA-256), `repInfo(for:)` carries the flag, and because the stream layer is offer-agnostic the receiver re-tags the delivered rep from the offer (`pull`) and the window extracts the `.aar` into a real folder (`stagedFileURL`, under a double-disk free-space guard) so a Finder paste recreates the tree. Mixing files and folders in one copy just works ‚Äî each selected item is its own rep. Same-named files in one copy get distinct staged URLs from `ClipboardFileStaging` (it uniquifies a repeated name with a ` (n)` suffix), and an offer is capped to `ClipboardContent.maxOfferableRepresentations` (0xFFFF) so a `transfer_id`'s 16-bit rep index can't alias. The eager `InboundCollection`/`onTransferComplete`/`commitInbound` assembly is retired; the channel-wide receiver `onComplete`/`onAbort` is now a log-only fallback for an unexpected unawaited transfer. Stale generations are detected by both sides (`AtomicGeneration` mirrors the current outbound generation for the sender's off-actor supersession check) so a transfer that races a newer offer aborts rather than overwriting fresher state. A `ClipboardRequest` the receiver can't answer ‚Äî stale generation, out-of-range rep index, or UTI mismatch ‚Äî is answered with a `ClipboardStreamAbort` (`ClipboardStreamSender.rejectRequest`) rather than silently dropped, so a peer's parked lazy pull wakes immediately instead of stalling to its `lazyPullTimeout` backstop (#357). Offer dedup and echo suppression compare 32-byte `ClipboardContent.digest` values, never retained copies (a `.file` rep folds in its streamed sha256; a `.pendingRemote` placeholder uses a distinct digest domain tag). A transfer the requester can't accept ‚Äî or a volume that fills mid-stream ‚Äî fails as a `.diskFull(needed:available:)` `ClipboardTransferIssue`, surfaced (like clipboard-scoped peer error frames) through the observable `lastTransferIssue` for the window's transient-message strip. The service is constructed lazily by the vsock clipboard listener when the guest connects, so `VMInstance.clipboardService` may be nil until then. No `Hello` is exchanged on this channel ‚Äî version/liveness and the `clipboard.stream.v1` capability gate live on the always-on control channel (`VsockControlService`); the service only runs when the peer advertised that capability, so there is no legacy single-frame fallback. The O(payload) digest work runs off the main actor via `ClipboardContent.makeOffActor`, and per-transfer chunk I/O runs on the sender's/receiver's own serial queues so a big copy/paste never blocks the UI; the host uses `@MainActor` `async`/`await` for its pulls (no blocking), while the guest agent blocks its main thread (woken off-main ‚Äî see below). (Giant *inline*, non-file content still touches the main thread for the unavoidable `NSPasteboard` read/write.)

- **`VsockControlService`** ‚Äî `@MainActor` `@Observable` consumer of the always-on control channel (`KernovaVsockPort.control = 49154`). Sends a host `Hello` on start; processes the guest's `Hello` to populate `agentVersion` and `isConnected`; emits and consumes bidirectional `Heartbeat` frames on a configurable cadence (default 5 s). A liveness watchdog flips `agentStatus` to `.unresponsive(version:)` after `unresponsiveAfter` (default 15 s) of inbound silence, and tears down the channel after `terminateAfter` (default 30 s) ‚Äî letting the listener accept a fresh connection. `agentStatus` classifies the guest-reported `Hello.agent_info.agent_version` against `KernovaGuestAgentInfo.bundledVersion` via `AgentStatus.isObservedVersionCurrent(_:bundled:)`, which delegates to `KernovaVersionComparison.isAtLeast` in the `KernovaKit` package ‚Äî the single numeric comparator (`String.compare(_:options: .numeric)`, correct for dotted-decimal SemVer like `0.9.0` vs `0.10.0`) shared by the host's `agentStatus` / `VMInstance.recordObservedAgentVersion` auto-eject decision *and* the guest's own menu update line, so all sites agree on current-vs-outdated; a `bundledAgentVersion` initializer parameter lets tests vary the comparison target. The host's `Hello` also carries `bundled_agent_version` (the version this host ships) host‚Üíguest, so the guest can label its own update state; the guest leaves the field empty in its Hello and an empty value reads as "unknown" (no update prompt). The listener is installed unconditionally for every macOS guest with a `VZVirtioSocketDevice`, so `agentStatus` is meaningful even when clipboard sharing is disabled. The host `Hello` advertises `KernovaCapability.controlChannelDefaults` (control, heartbeat, and `clipboard.stream.v1`); the guest's `Hello` capability list is inspected and the `PolicyUpdate` clipboard bit is gated on the guest advertising `clipboard.stream.v1` (mirrored on the guest side) ‚Äî an agent that can't stream keeps clipboard disabled and the host shows the existing `.outdated` "update the guest agent" affordance. Also delivers the per-VM toggle policy to the guest: an optional `policyProvider: () -> AgentPolicySnapshot` closure is read on every guest `Hello` and the resulting snapshot is sent as a `PolicyUpdate` frame so the guest agent knows which capabilities (log forwarding, clipboard sharing) to enable. `sendPolicyUpdate(_:)` is also public so runtime hot-toggle pushes can reuse the same path. An optional `onAgentVersionObserved: (String) -> Void` closure fires once per guest `Hello` carrying a non-empty `agent_version`; `VMInstance.startVsockServices` wires it to `recordObservedAgentVersion(_:)`, which persists the new value via `onUpdateConfiguration` (routed through the view-model's `updateConfiguration` dispatcher) and clears the post-start watchdog state.

- **`AgentPolicySnapshot`** ‚Äî Plain `Equatable` `Sendable` struct (`logForwardingEnabled`, `clipboardSharingEnabled`) carrying the pair of toggles the guest agent honors. Decouples `VsockControlService` from `VMConfiguration`: the host injects a closure that reads the latest config when policy is sent, so reconnects always pick up the current snapshot.

- **`VsockListenerHost`** ‚Äî `@MainActor` wrapper around `VZVirtioSocketListener` bound to one vsock port. The nonisolated `VZVirtioSocketListenerDelegate` callback dups the connection's file descriptor, sets `SO_SNDTIMEO` defensively (a belt-and-suspenders backstop ‚Äî the streaming credit window is the real write bound), and bridges back to MainActor before constructing a `VsockChannel` and handing it to a caller-supplied closure. One instance per service; multiple coexist on the same `VZVirtioSocketDevice`.

- **`VsockGuestLogService`** ‚Äî `@MainActor` consumer that owns one accepted `VsockChannel` for the lifetime of a guest connection. Forwards `LogRecord` frames through a `GuestLogEmitter` abstraction (default `OSLogGuestLogEmitter`, subsystem `app.kernova.guest`); guest log levels map 1:1 onto `os.Logger` methods. `Error` frames go to the host's own diagnostic logger; `Hello` / `Heartbeat` payloads on this port log a wrong-port warning (those belong on the control channel). The service self-terminates on EOF.

- **`VsockPorts`** ‚Äî Central registry of port assignments (`KernovaVsockPort.control = 49154`, `KernovaVsockPort.clipboard = 49152`, `KernovaVsockPort.log = 49153`) so each service gets its own listener on a distinct port instead of in-band multiplexing. The control listener is always installed for macOS guests with a `VZVirtioSocketDevice`; the log listener is gated on `configuration.agentLogForwardingEnabled`; the clipboard listener is gated on `configuration.clipboardSharingEnabled`. Both gates are also re-evaluated at runtime via `VMInstance.applyLivePolicy(oldConfig:newConfig:)` ‚Äî flipping the toggle while a macOS VM is running installs or tears down the listener and pushes a fresh `PolicyUpdate` to the guest agent. Linux clipboard sharing is restart-only (the SPICE port must be declared at config-build time).

**Storage topology mirrors VZ.** `VMConfiguration` carries two ordered lists that map directly onto VZ's two storage surfaces. `storageDisks: [StorageDisk]?` maps onto `vzConfig.storageDevices`; position [0] boots first on EFI guests. Each entry's `kind` (`.virtio` or `.usbMassStorage`) is inferred from the file extension at add-time via `StorageDisk.defaultKind(forPath:)` ‚Äî `.iso`/`.dmg` default to USB mass storage so installer media doesn't shift the main disk's `/dev/vda` letter when reordered for boot, everything else defaults to virtio. `removableMedia: [RemovableMediaItem]?` maps onto `usbControllers[0].usbDevices` ‚Äî hot-pluggable, no boot semantics. The same bundled-disk entry (`Disk.asif`, internal, virtio) appears in `storageDisks` as a regular row; nothing in the data model singles it out as "the main disk." Each `StorageDisk` carries a user-editable `label` (renamed inline or via the row context menu through `VMLibraryViewModel.renameStorageDisk`; cosmetic ‚Äî the backing file keeps its stable UUID name and the guest is unaffected); new disks get a collision-free default label via `StorageDisk.uniqueLabel`. The model stores **no** physical sizing ‚Äî on-disk footprint, virtual capacity, and creation date are read live from the file at display time (uniformly for the main disk and additional disks), since a size can change out-of-band (e.g. a CLI resize) and any stored copy would go stale. The subtitle reads run **off the main thread** via `populateDiskSubtitle` (placeholder ‚Üí detached read ‚Üí fill-in; reads are cancelled when a field is re-bound); Get Info also reads off-main (a detached size + creation-date read, then presents). Both attachment lists share one `VMSettingsViewController` implementation ‚Äî `refreshAttachmentList`/`makeAttachmentRow` (rebuild vs. in-place), one `buildAttachmentContextMenu` with eight `@objc` handlers dispatched on an `AttachmentRef(kind, id)` `representedObject` (Eject is removable-only, so its handler is a no-op for storage refs), `presentAttachmentInfoPopover`, and `commitAttachmentRename` ‚Äî parameterized by `AttachmentKind` rather than duplicated per list. **Removable media (`RemovableMediaItem`) is at full parity with storage disks**: same editable `label` (inline rename or context menu, via `VMLibraryViewModel.renameRemovableMedia` ‚Äî safe to rename while running, since a label-only edit leaves `path`/`readOnly` untouched and the reconcile only detaches/reattaches on those, so the medium stays mounted), the same `AttachmentRowView` UI **plus an inline trailing Eject button that storage rows omit** (a one-click detach ‚Äî no confirmation, file untouched, via `removeRemovableMedia(_:from:trashFile: false)` ‚Äî hot-pluggable media is swapped often, so it earns a dedicated control; the eject glyph is shared with the Shared Directories row button), the right-click menu (Rename / Get Info / Show in Finder / Copy Path / Copy File Name / Read Only / **Eject** / Remove ‚Äî Eject, the no-confirmation detach, is removable-only, the one menu item storage rows lack), the same `AttachmentInfoPopoverContentViewController`, and the same live on-disk/allocated sizing (always external ‚Äî a raw `.iso`/`.dmg` reports its logical size as the capacity).

**Live-editable fields and their dispatch.** `VMConfiguration.liveEditableFieldsChanged(old:new:)` is the single source of truth for "did anything change that should take effect while the VM is running?" It combines `hotToggleFields` (a typed `[KeyPath<VMConfiguration, Bool>]`) with `removableMediaChanged(old:new:)`, which array-compares the `removableMedia` lists. `storageDisks` changes are deliberately NOT live-editable ‚Äî they go to `vzConfig.storageDevices`, which VZ requires fixed at start time, so the settings UI keeps that section locked while the VM is running. `VMSettingsViewController` mutates via the centralized `VMLibraryViewModel.updateConfiguration(of:mutate:)` dispatcher (persist + `applyLivePolicy`). Every control's target/action handler routes its write through this dispatcher; no settings control writes to `instance.configuration` directly.

**Removable-media reconcile.** `VMLibraryViewModel.applyLivePolicy(for:old:new:)` forks: vsock listener changes go through `VMInstance.applyLivePolicy`; `removableMedia` changes are dropped into `pendingRemovableMediaTarget` and a coalesce-and-drain task (`runRemovableMediaReconciliation`) calls `applyLiveRemovableMediaChange(for:target:)` until the pending dictionary empties. The reconciler computes a per-id diff against `instance.liveRemovableMedia` (add / remove / mutate-in-place / reorder-noop), detaches first to avoid duplicate-UUID conflicts on swaps, then attaches. `deviceNotFound` (guest-side ejection) and `noVirtualMachine` (VM torn down) are handled distinctly; on any other framework error the reconciler calls `reconcileConfigToLiveState(for:lookup:)` to rebuild `config.removableMedia` from `instance.liveRemovableMedia` ‚Äî so the UI snaps to what's actually attached instead of describing a state VZ refused. The rollback bypasses `updateConfiguration` (direct write + `saveConfiguration`) to avoid re-entering the reconcile pipeline.

**Save-state device UUID persistence.** `VZUSBDeviceConfiguration.uuid` is matched against the saved-state file's recorded device list during `restoreMachineStateFrom(url:)` ‚Äî fresh UUIDs each launch would break restore. `RemovableMediaItem.id` becomes `VZUSBMassStorageDeviceConfiguration.uuid` and is persisted with the entry in `config.json`. For virtio entries in `storageDisks`, the entry's `id` is also used as the `VZVirtioBlockDeviceConfiguration.blockDeviceIdentifier` (truncated to 20 ASCII chars), giving Linux guests a stable `/dev/disk/by-id/virtio-<identifier>` symlink ‚Äî with the exception of the bundle's primary disk (`Disk.asif`, identified by `ConfigurationBuilder.isMainBundleDisk(_:layout:)`), which intentionally has no `blockDeviceIdentifier` set so the pre-refactor `/etc/fstab` behavior is preserved. The synthesized default main disk (used when `storageDisks` is nil/empty) derives its UUID deterministically from `SHA256(bundleURL.path)` via `ConfigurationBuilder.stableMainDiskID(forBundleAt:)`, so entry-by-id lookups (notably `removeStorageDisk` and the settings list's read-only/delete row actions) are stable before the user has materialized the list. `clonedForNewInstance` regenerates every `StorageDisk.id` and `RemovableMediaItem.id` so two bundles don't share device identity.

All service implementations conform to protocols defined in `Services/Protocols/`. This enables full dependency injection ‚Äî tests use mock implementations that track call counts and support error injection.

### ViewModels

**Files:** `VMLibraryViewModel.swift`, `VMLifecycleCoordinator.swift`, `VMCreationViewModel.swift`, `VMDirectoryWatcher.swift`

- **`VMLibraryViewModel`** is the central `@Observable` view model. It owns the array of `VMInstance`s and handles list-level operations: add, remove, rename, reorder, selection tracking. VM order is user-customizable via drag-and-drop in the sidebar, persisted as a UUID array in `UserDefaults` (key `"vmOrder"`). VMs not in the custom order (newly created/discovered) sort after ordered VMs by `createdAt`. For lifecycle operations (start, stop, install), it delegates to `VMLifecycleCoordinator`. Clone and import operations use a "phantom row" pattern: a `VMInstance` with `isPreparing = true` appears immediately in the sidebar with a spinner while the file copy runs asynchronously via `Task.detached`. The `hasPreparing` computed property enforces serialization ‚Äî only one clone/import at a time. Cancellation removes the phantom row, cancels the task, and cleans up partial files on disk. Force-stop is surfaced via `confirmForceStop()` which presents a confirmation dialog. **All `VMConfiguration` mutations route through `updateConfiguration(of:mutate:)`** ‚Äî a single dispatcher that takes an `inout` closure, no-ops when the closure produces an equal value, persists via `saveConfiguration`, and calls `applyLivePolicy(for:old:new:)`. This is the only place persist + live-policy fire together; settings-UI bindings (`configBinding`, `storageDiskBinding`, `removableMediaBinding`), install/uninstall flows, rename, display-window callbacks, and guest-driven `VMInstance.onUpdateConfiguration` writes all funnel through it.

- **`VMLifecycleCoordinator`** is an `@MainActor` coordinator that owns the lifecycle services (`VirtualizationService`, `MacOSInstallService`, `IPSWService`). It orchestrates multi-step operations like macOS installation (which involves IPSW download ‚Üí platform file creation ‚Üí VM configuration ‚Üí installation). This separation keeps `VMLibraryViewModel` focused on list management. The coordinator enforces **per-VM operation serialization** ‚Äî at most one lifecycle operation can be in flight for a given VM at any time; concurrent requests are rejected with `VMLifecycleCoordinator.LifecycleError.operationInProgress`. `stop` and `forceStop` bypass serialization entirely (clearing the active-operation token before calling the service) so users can always cancel hung operations.

- **`VMCreationViewModel`** drives the multi-step creation wizard. It tracks the current step, validates inputs at each stage, and produces a `VMConfiguration` + disk image on completion. It is a pure `@Observable` state machine with no UI framework dependency ‚Äî the wizard's pure-AppKit step view controllers (`Views/Creation/`) read/write it directly and the shell observes it via `observeRecurring`.

- **`VMDirectoryWatcher`** uses `DispatchSource.makeFileSystemObjectSource` to monitor the VMs directory for external changes (e.g., a user restoring a VM from Trash via Finder). When changes are detected, it triggers reconciliation in `VMLibraryViewModel` to sync the in-memory list with disk.

- **`SystemSleepWatcher`** (see Services section) is also owned by `VMLibraryViewModel`, triggering `pauseAllForSleep()` and `resumeAllAfterWake()` on system sleep/wake events. Auto-paused VMs are tracked in `sleepPausedInstanceIDs` so user-paused VMs are not accidentally resumed.

## Key Design Decisions

### 1. Pure-AppKit UI

**What:** The app is entirely AppKit. `NSSplitViewController` owns the sidebar/detail layout, `NSToolbar` with native `NSToolbarItem`s the toolbar, and `NSWindow` window management. The sidebar is a source-list `NSOutlineView` (`SidebarViewController`); the detail pane is an empty-state view ‚áÜ `VMDetailRouterViewController` (`VMSettingsViewController` + placeholders + install progress). The VM display (`VZVirtualMachineView`) is managed by `VMDisplayBackingView` ‚Äî in the detail pane `DetailContainerViewController` layers it on top of the detail content, and in pop-out/fullscreen windows `VMDisplayWindowController` uses it directly as the window's content view. The app target imports no SwiftUI.

**Why:** The app needs precise control over native macOS chrome ‚Äî toolbar items, split-view behavior, sidebar collapsibility, popover/sheet anchoring ‚Äî and Swift 6 strict-concurrency-friendly observation. SwiftUI's `NavigationSplitView`/`.toolbar` and the AppKit‚ÜîSwiftUI bridge boundaries proved fragile for this; the project incrementally converted every surface (sidebar, wizard, popovers/alerts, then the detail pane) to AppKit. Toolbar state is validated via `NSToolbarItemValidation`, observation is centralized in `observeRecurring`, and there are no hosting-bridge quirks.

**Alternatives:** SwiftUI `NavigationSplitView` with `.toolbar` modifiers ‚Äî simpler but encountered persistent toolbar layout issues. A hybrid AppKit-shell / SwiftUI-content app (the prior architecture) ‚Äî workable, but the bridge shims (sheet/popover anchors, `WindowAccessor`) added complexity that pure AppKit removed.

### 2. VM bundle as `.kernova` package directory

**What:** Each VM is a directory with a `.kernova` extension containing `config.json`, the disk image, auxiliary storage, save files, and serial logs.

**Why:** Treats each VM as an atomic unit in Finder. Users can move, copy, or delete VM bundles as single items. The directory structure is predictable via `VMBundleLayout`, and `config.json` makes the format human-inspectable.

**Alternatives:** SQLite database, single directory with UUID-named files, or Core Data. The bundle approach is simpler and more transparent.

### 3. ASIF disk images via bundled templates

**What:** Disk images use Apple Sparse Image Format (ASIF), a macOS 26 format. Pre-built templates are stored lzfse-compressed in `DiskTemplates/` (~3 KB each) and decompressed at VM creation time.

**Why:** ASIF provides near-native SSD performance with space efficiency ‚Äî a 100 GB disk image starts at ~4 MB on disk and grows as the guest writes data. The template approach is fully sandbox-safe (no subprocess spawning), unlike the previous `hdiutil`-based approach.

**Alternatives:** Raw disk images (simple but waste space), QCOW2 (not natively supported by Virtualization.framework), or runtime `hdiutil` invocation (not sandbox-safe, requires process spawning).

### 4. `@MainActor` isolation strategy

**What:** Everything touching `VZVirtualMachine` is `@MainActor`-isolated. `VMInstance`, `VirtualizationService`, and `MacOSInstallService` are all `@MainActor`. Services that don't touch VZ are `Sendable` structs with no mutable state.

**Why:** `VZVirtualMachine` is main-thread-only (Apple requirement). Swift 6 strict concurrency enforces this at compile time. Making the boundary explicit prevents accidental cross-thread access. `Sendable` structs for stateless services avoid unnecessary main-thread bottlenecks.

**Bridging:** Some `VZVirtualMachine` delegate callbacks arrive on the main thread but aren't marked `@MainActor` in the API. These use `nonisolated(unsafe)` with `MainActor.assumeIsolated` to bridge back.

**Alternatives:** Wrapping all VZ access in `MainActor.run {}` calls ‚Äî more boilerplate and easier to miss a call site.

### 5. Service protocol abstraction

**What:** Every service has a corresponding protocol (`VirtualizationProviding`, `VMStorageProviding`, etc.) defined in `Services/Protocols/`.

**Why:** Enables dependency injection for testing. Mock implementations can track call counts, return canned responses, and inject errors via `throwError` properties. The coordinator and view models accept protocol types, not concrete implementations.

**Alternatives:** No protocols, test against real services ‚Äî would require actual VM operations in tests, making them slow and environment-dependent.

### 6. VMLifecycleCoordinator separation

**What:** `VMLifecycleCoordinator` sits between `VMLibraryViewModel` and the services, orchestrating multi-step operations. It also enforces per-VM operation serialization ‚Äî a token-based `[UUID: UUID]` dictionary maps each VM to its current operation token and rejects concurrent requests with `VMLifecycleCoordinator.LifecycleError.operationInProgress`. `stop`/`forceStop` bypass serialization entirely ‚Äî they clear the token *before* calling the service, which invalidates any in-flight operation's `defer` guard and prevents stale removals.

**Why:** macOS VM installation is a multi-step process (download IPSW ‚Üí create platform files ‚Üí configure VM ‚Üí install). Putting this in `VMLibraryViewModel` would bloat it with orchestration logic. The coordinator keeps the view model focused on list management and selection, while the coordinator handles operational complexity. Operation serialization prevents undefined behavior from concurrent `VZVirtualMachine` calls (e.g., double-start or pause-during-start).

**Alternatives:** Fat view model (simpler structure but harder to test and maintain), or individual operation objects (more granular but more types to manage).

### 7. Native NSToolbar with observation-driven validation

**What:** The main window and display window use `NSToolbar` with `NSToolbarDelegate` creating native `NSToolbarItem`s. Shared toolbar items (lifecycle, suspend, clipboard, display, settings toggle) are managed by `VMToolbarManager`, a `@MainActor` `NSObject` subclass that handles item creation, state updates, and action routing for both controllers. Each controller configures it with an `instanceProvider` closure and a `Configuration` struct that captures per-controller differences (identifier strings, `isPreparing` checks, display capability gating, presence of the settings toggle). Toolbar state is driven by the shared `observeRecurring` helper (see Utilities), directly setting `isEnabled` on subitems on change. All toolbar items ‚Äî both `NSToolbarItemGroup`s and plain `NSToolbarItem`s whose enabled state is driven manually ‚Äî use `autovalidates = false`, because `NSToolbarItemValidation` returns `true` for any shared identifier when a VM is selected, and AppKit's autovalidation would otherwise fight the observation-driven `isEnabled` writes and produce a visible flicker on selection changes. When an item swaps its image/label based on state (Start‚ÜîResume, Pop Out‚ÜîPop In, Show‚ÜîHide Settings), the mutation is guarded behind a label-equality check so no-op updates don't trigger an AppKit redraw.

Both toolbars are user-customizable with autosaved configurations (#319). The manager's state-update lookups treat an absent item as a legitimate silent skip (the user may have removed it via customization) and warn only for present-but-malformed groups. Because palette-added items are created in factory-default state and `autovalidates` is off, both controllers implement `toolbarWillAddItem` and refresh item state one runloop turn later. The main toolbar's identifier is `KernovaMainToolbar`; every VM's display window shares the single `KernovaVMDisplayToolbar` identifier so AppKit synchronizes customization across display windows and persists one configuration (window *frames* remain autosaved per-VM). The display window auto-hides its toolbar in fullscreen via the `.autoHideToolbar` presentation option (`window(_:willUseFullScreenPresentationOptions:)`) rather than toggling `toolbar.isVisible`, which would contaminate the autosaved configuration.

**Why:** Native `NSToolbarItem`s provide reliable layout, proper `.sidebarTrackingSeparator` support, and standard macOS toolbar appearance. The `observeRecurring` helper handles re-registration after each change and `[weak self]` teardown uniformly across every window controller, so toolbar updates stay reactive without SwiftUI and without duplicating the observation-loop boilerplate at each site. The shared `VMToolbarManager` eliminates ~150 lines of duplicated toolbar logic between `MainWindowController` and `VMDisplayWindowController`, ensuring toolbar changes are applied in one place.

**Alternatives:** SwiftUI `.toolbar` modifiers on a hosting controller ‚Äî simpler declarative API but caused persistent layout issues with grouped items and sidebar tracking.

### 8. Apple Silicon-only (arm64)

**What:** The project sets `ARCHS = arm64` at the project level ‚Äî no Intel (x86_64) slice is built, and the codebase contains no `#if arch(arm64)` conditionals.

**Why:** Virtualization.framework gates the macOS-guest APIs (`VZMacOSInstaller`, `VZMacOSRestoreImage`, `VZMacAuxiliaryStorage`, recovery start options) and VM save/restore (`saveMachineStateTo`/`restoreMachineStateFrom`) to Apple Silicon, so an Intel build would lose macOS guests *and* suspend/restore ‚Äî and with a macOS 26 deployment target (the final Intel macOS release), the addressable Intel hardware is a handful of models on their last OS. The guards previously kept for a hypothetical Intel slice were never compiled (all dev and CI is arm64 Debug with `ONLY_ACTIVE_ARCH`), and the slice had silently stopped building.

**Alternatives:** Keep standard architectures with `#if arch(arm64)` guards and ship a Linux-guests-only Intel app ‚Äî rejected as untested, already broken, and not worth ~38 conditional-compilation sites.

Four standalone targets are built alongside the main app ‚Äî two CLI tools, one app extension, and one unit-test bundle:

- **KernovaQuickLook** ‚Äî Quick Look preview extension embedded in `Contents/PlugIns/`. - **KernovaGuestAgent** ‚Äî Not embedded directly. A menu-bar (`.accessory`, `LSUIElement`) app bundle that runs inside macOS VMs ‚Äî an `NSStatusItem` dropdown (no Dock icon, no window) gives the guest a place to show status, built by `GuestAgentAppDelegate` (`@main`) + `GuestAgentStatusItemController`. The dropdown leads with live status and offers About + Quit (clipboard/log enablement is host-driven, so it reflects state rather than offering switches): a live host-connection line (`HostConnectionState` pulled from `VsockGuestControlAgent` on menu open; the status-item icon also tracks it live via an `onStateChange` push), a "Status" submenu grouping the two host-driven capability states ‚Äî a Log Forwarding line (`Log Forwarding: enabled/disabled`, read from `VsockHostConnection.isLogForwardingEnabled`) then a Clipboard line (`ClipboardActivity`: the feature-state pair `enabled`/`disabled` plus a symmetric 2√ó2 of last *flow* events ‚Äî `offeredToHost`/`offeredFromHost` (someone copied) and `sentToHost`/`receivedFromHost` (someone pasted); the four flow cases are a last-event signal because the inbound pull blocks the main thread so an in-flight transfer can't be drawn, and `sentToHost` is marked at outbound-stream start rather than completion), an "About Kernova Guest Agent" item that opens the standard AppKit About panel (`aboutTapped` ‚Üí `NSApp.orderFrontStandardAboutPanel`, mirroring the host app's `showAboutPanel`; icon + name + version/build + copyright all come from the bundle's `Info.plist`), and Quit (a clean `exit(0)` that the LaunchAgent's `KeepAlive={SuccessfulExit:false}` leaves quit). Identity and version/build moved out of the menu into that About panel; only when the host bundles a newer agent does a top-level "Update available ‚Äî host bundles X" hint appear above the connection line and Status submenu (and as the About panel's credits), derived from the host's bundled version (see below). The pure text mappers live in `GuestAgentMenuText`. The agent maintains three long-lived vsock connections to the host: an always-on control channel (`VsockGuestControlAgent` on port 49154 carrying the version handshake, bidirectional heartbeat, and the inbound `PolicyUpdate` push), log forwarding (`VsockHostConnection` on port 49153), and bidirectional clipboard sync (`VsockGuestClipboardAgent` on port 49152). All three connections are independent ‚Äî a disconnect on one doesn't take the others down ‚Äî and all share a `VsockGuestClient` helper that owns the connect/retry/serve loop. The log + clipboard agents start in a default-disabled state (paused at the `VsockGuestClient` level so no connect attempts run); they're enabled when the control agent receives the host's first `PolicyUpdate` and routes it via an `onPolicy` closure to `vsockConnection.setEnabled(_:)` and `clipboardAgent.setEnabled(_:)`. Disabling either capability discards buffered records and pauses the reconnect loop, so the guest stops generating traffic rather than relying on the host to ignore it. It enables clipboard only when the host's `Hello` advertised `clipboard.stream.v1` (the capability gate mirrors the host's), so an agent talking to a host that can't stream stays off, and the agent's `MARKETING_VERSION` is bumped in lockstep (0.15.0 ‚Üí 0.18.0 with the streaming protocol; ‚Üí 0.19.0 for the lazy-paste + runloop change; ‚Üí 0.20.0 for the symmetric dropped-request reject, #357; ‚Üí 0.21.0 for multi-file copy in both directions; ‚Üí 0.22.0 for folder copy via in-process AppleArchive; ‚Üí 0.23.0 for the menu-bar app-bundle repackage; ‚Üí 0.23.1 for the app icon; ‚Üí 0.23.2 for the Login-Items bundle association; ‚Üí 0.23.3 for the identity-signed agent (Team ID, so Login Items resolves the developer/icon); ‚Üí 0.23.4 for the spaced product name (`Kernova Guest Agent.app` / `CFBundleName`, so Finder and Login Items show "Kernova Guest Agent" rather than "KernovaGuestAgent"); ‚Üí 0.24.0 for the About-panel menu restructure (identity/version moved into the standard About panel, dropdown leads with live status); ‚Üí 0.25.0 for the org.nspasteboard.* marker disposition (transient/auto-generated snapshots suppressed, concealed ones flagged with the offer's `is_concealed`); ‚Üí 0.26.0 for the menu restructure (symmetric clipboard flow states, a Log Forwarding line, and a "Status" submenu grouping both capabilities); ‚Üí 0.27.0 for inline reps spilling to disk + mmap above `maxResidentInlineBytes` (no inline size cap)) so an out-of-date host/guest pair surfaces the existing "update the guest agent" affordance. The clipboard agent polls `NSPasteboard.general` at 500 ms (changeCount gate first), snapshots the first item's UTI-tagged representations ‚Äî filtered by the shared `ClipboardSnapshotPolicy` (transient `org.nspasteboard.*` markers and file references never cross the wire; **no size cap** ‚Äî every transfer is chunk-streamed; a snapshot-level disposition suppresses a transient/auto-generated snapshot wholesale and flags a concealed (password) snapshot with the offer's `is_concealed`) ‚Äî and announces changes to the host via a **metadata-only** `ClipboardOffer` carrying a `ClipboardRepresentationInfo` (`uti`, `byteCount`, `filename`, `isInline`, `isDirectory`) per representation. **Inbound paste is lazy.** On a host offer the agent registers one `NSPasteboardItem` *promise* per item ‚Äî one shared inline item plus one per file rep ‚Äî each backed by its own `LazyClipboardDataProvider` (`writeItems`), and pulls **no bytes**: an inline rep promises its content UTI, each file rep promises `public.file-url` (an image file also its image UTI), and each item's provider resolves a requested type to *its own* offer-rep index, so several files' `.fileURL` pulls each resolve to their own file instead of collapsing to the first. The post-write `changeCount` is captured immediately so the poll never reads ‚Äî and thereby self-triggers ‚Äî the agent's own promise (echo suppression at promise time). When the guest OS asks for a promised type, the provider's `provideData` runs synchronously on the agent's main thread, sends one `ClipboardRequest`, **blocks** on `LazyPullCoordinator.pull` while the host streams `ClipboardStreamBegin`/`ClipboardChunk`/`ClipboardStreamEnd` (via `ClipboardStreamSender`/`ClipboardStreamReceiver` over the shared `KernovaKit` engine), and is woken **off-main** by the receiver's per-`transfer_id` `awaitTransfer` hook (never a main-thread hop, which the blocked main thread would deadlock). For these CFRunLoop-delivered `provideDataForType` callbacks to fire at all, the agent runs `NSApplication.run()` (the menu-bar app's run loop is a strict superset of the bare CFRunLoop ‚Äî and of `dispatchMain()`, which drains only the GCD main queue), so the lazy-paste promises *and* the existing `.main` DispatchSource poll/signal handlers keep working. (`.accessory` apps are also App-Nap candidates, so the agent holds an `NSProcessInfo` activity while clipboard sharing is enabled to keep the 0.5 s poll from being throttled.) Each rep is pulled at most once and cached, so an image promised as both its UTI and `public.file-url` is fetched a single time. On the receive side, a small inline representation reassembles in RAM while a large inline one (past `maxResidentInlineBytes`) or a file representation streams to a local temp file under a free-space guard (`ClipboardFileStaging.makeSink`), a spilled inline rep being mmapped back so inline content has no size cap; a too-large file fails the pull's free-space pre-flight and the provider returns empty without a request, and a mid-flight or pre-Begin abort/teardown/timeout wakes the blocked pull and returns empty (the host Aborts every request it drops, so a supersession-mid-paste race wakes the pull immediately rather than parking to the 120 s backstop ‚Äî #357). A non-image file rep resolves its `public.file-url` to the staged temp file; an image file rep is served inline for the image UTI and *also* as a staged file-url (so Notes shows the image and Finder can paste it) ‚Äî the wire only ever carries the filename, never a host path. A superseding offer, a host `ClipboardRelease`, or a connection teardown retracts the promise and `failAll()`s the coordinator so any blocked provider returns immediately (the eager `InboundCollection`/`onTransferComplete`/`finishInbound`/`applyPairs`/`inboundStagingQueue` assembly is retired; the channel-wide receiver `onComplete`/`onAbort` is now a log-only fallback). Copied *files* (outbound; Finder ‚åòC puts one `public.file-url` + name per item, not the bytes) are offered from a `stat` only ‚Äî one disk-backed `.file` representation per file (read via `itemFileURLs`, each skipped individually if it's already inside the agent's own staging root), never read at offer time, no size cap, streamed 64 KiB at a time on request ‚Äî the guest-side mirror of the host window's file-drop expansion. A copied **folder** is detected via `.isDirectoryKey` and archived **eagerly off the main queue** (`archiveAndOffer` hops to a global queue and back; an `archiveInFlight` flag stops overlapping 0.5 s polls double-archiving, and the completion drops itself if the clipboard changed mid-archive) into one `isDirectory` `.aar` rep staged in a dedicated `agent-send` `ClipboardFileStaging` (kept separate so an outbound archive's generation can't collide with an inbound transfer's). Inbound, a directory offer promises only `.fileURL`; `provideData` extracts the streamed `.aar` (re-tagged `isDirectory` from the offer in `pullRepresentation`) into a real folder under the staging root (`reserveDirectory`, double-disk free-space-guarded), cached per rep so a repeated pull reuses the extracted tree. Echo suppression is digest-based (`ClipboardContent.digest`, a `.file` rep folding in its streamed sha256), reset on reconnect so a fresh host instance gets re-offered the current clipboard. The agent depends on the local `KernovaKit` SPM package for the wire types and channel implementation. Packaged into a disk image at build time by the "Package Guest Agent DMG" Run Script build phase, which `ditto`-copies the built `Kernova Guest Agent.app` (preserving its code signature) into the image. The disk image (containing the `.app` bundle, `install.command`, `uninstall.command`, and a LaunchAgent plist) is placed in `Contents/Resources/KernovaGuestAgent.dmg`. At runtime, a single Virtual Machine-menu item (`toggleGuestAgentDisk`) manages this disk for the active VM. While the disk is **not** attached the item attaches it (mounting it as USB mass storage) with a purpose-framed title that hints at *why*, driven by `agentStatus`: Install (`.waiting`) / Update (`.outdated`) / Reinstall (`.expectedMissing`) / Manage (`.current` or `.unresponsive` ‚Äî the agent is present but the user may want to reinstall or run `uninstall.command`). `.unresponsive` is treated like `.current` because it persists when the user disables/uninstalls/kills the agent in the guest; only the genuinely-transient `.connecting` leaves the item disabled. While the disk **is** attached the same item switches to **Eject Guest Agent Media** (a detach with no file deletion). All states require a live VM (`canAttachUSBDevices`). The mount purpose drives the post-attach instructions alert (`presentInstallerMounted(vmName:purpose:)`): `.install` points at `install.command`, `.manage` names both `install.command` and `uninstall.command`. The guest user runs `install.command`, which `ditto`-copies the `.app` to `~/Applications` (strips quarantine, cleans up any legacy bare-binary install) and registers a user-space LaunchAgent that execs the bundle's inner binary; once the agent handshakes a current version the host **auto-ejects** the disk (see `VMInstance.onAgentBecameCurrent`). Eject is intentionally a menu-bar / Settings affordance only ‚Äî it is *not* added to the sidebar agent-status popover or clipboard window, which stay install-focused. The vsock reconnect loop uses a flat 5s retry interval; `SO_RCVTIMEO` / `SO_SNDTIMEO` are set to 30 s as a safety net against wedged read/write calls. The build number is injected via `INFOPLIST_PREPROCESS`: a pre-Sources build phase ("Set Build Number from Git") writes `#define AGENT_BUILD_NUMBER` (set to the git commit count scoped to `KernovaGuestAgent/`) to a header in `DERIVED_FILE_DIR`, and the explicit `Info.plist` references that macro for `CFBundleVersion`. As an app bundle the preprocessed plist is the bundle's real `Contents/Info.plist` (the bare-tool `CREATE_INFOPLIST_SECTION_IN_BINARY` embedding is no longer used), and `Bundle.main` reads version/build from it. The bundle carries an app icon (`Assets.xcassets` with an `AppIcon` set sharing the host app's icon; `ASSETCATALOG_COMPILER_APPICON_NAME = AppIcon`) so it shows a proper icon in Finder and System Settings ‚Üí Login Items ‚Äî there is no Dock icon at runtime (`.accessory`). The product is named `Kernova Guest Agent` (`PRODUCT_NAME`), so the wrapper is `Kernova Guest Agent.app` and `CFBundleName` carries the spaces that surface in Finder and Login Items (neither honors `CFBundleDisplayName` for this ‚Äî Finder shows the wrapper filename, BTM shows `CFBundleName`); the executable and Swift module stay `KernovaGuestAgent` (pinned via `EXECUTABLE_NAME` / `PRODUCT_MODULE_NAME`) so the LaunchAgent `ProgramArguments` path, the DMG build-phase inputs, and `install.command`'s `EXEC_REL` keep a space-free leaf. Because the synchronized-folder asset catalog only compiles into a target that has a `Resources` build phase, the target gained one when it became an application (the bare-tool form had none).
- **KernovaGuestAgentTests** ‚Äî Standalone unit-test bundle (no `TEST_HOST` / `BUNDLE_LOADER`) that covers the agent-side classes. Because `KernovaGuestAgent` is an application target (not a framework), its symbols are not linkable ‚Äî the test bundle instead compiles the agent's Swift source files directly (all except `GuestAgentAppDelegate.swift` ‚Äî the `@main` entry ‚Äî plus `Info.plist`, `Assets.xcassets`, and the shell scripts, excluded via `PBXFileSystemSynchronizedBuildFileExceptionSet`). This direct compilation makes all `internal` members accessible without `@testable import`, which is unavailable for tool targets. Three agent classes required light testability seams: `VsockGuestClient` gained a `socketProvider` closure injection point and a parameterized `retryInterval` (default unchanged at 5 s); `VsockGuestClipboardAgent` gained a `Pasteboard` protocol (with `NSPasteboard` conformance) and injected `client`/`pasteboard` init parameters; `VsockHostConnection` lifted `pendingLogs`, `lock`, `bufferFrame`, `flushPendingLogs`, and `logBufferLimit` from `private` to internal. Shared test helpers live in `TestHelpers.swift` (socket-pair factories, `waitUntil`, `nextFrame`, `awaitFirst`, `AtomicInt`, frame factories). Non-parallelizable in the scheme because each test worker loads the agent sources which include global state (`VsockLogBridge.connection`); tests share one runner process.

### Test Patterns

- **Framework:** Swift Testing (`@Suite`, `@Test`, `#expect`) ‚Äî not XCTest
- **Mocks:** 8 mock implementations conforming to service protocols, supporting call counting and error injection via `throwError` properties. Includes `SuspendingMockVirtualizationService` for testing operation serialization and `SuspendingMockUSBDeviceService` for testing the mount mutex in `mountGuestAgentInstaller` ‚Äî both suspend mid-operation to verify concurrent rejection. Rely on `@MainActor` cooperative scheduling (documented in the mocks) and enforce single-suspension via `precondition`
- **Factories:** Shared helpers like `makeInstance()`, `makeViewModel()`, `makeCoordinator()` reduce setup duplication
- **Error paths:** Mocks support setting `throwError` to inject failures and verify error handling
