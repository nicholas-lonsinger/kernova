# RELEASING.md

How to cut a **notarized Developer ID release** of Kernova. This is the only
distribution channel that exists today: the app already runs under the App
Sandbox in every configuration (#89 — see the *Mac App Store Readiness*
section of [CLAUDE.md](CLAUDE.md)), so TestFlight and the Mac App Store are a
matter of App Store Connect setup rather than further code work. Developer ID
("Direct Distribution" in Xcode's Organizer) remains the shipping channel; the
sandbox changes nothing in the flow below.

Releasing is **Nicholas-specific** and does not touch CI or the clone-and-run
story. CI builds Debug with `CODE_SIGNING_ALLOWED=NO`, and the Debug
configuration remains the profile-less path anyone can build against their own
team. Everything below concerns the **Release** configuration and the
Developer ID identity (team `8MT4P4GZL2`, bundle id `app.kernova`, Apple
Silicon only).

## One-time prerequisites

These are already set up; documented here so the environment is reproducible if
it ever has to be rebuilt (e.g. on a new Mac or after certificate expiry).

### 1. Developer ID Application certificate

Created via **Xcode → Settings → Accounts → Manage Certificates → + →
Developer ID Application** (requires the Account Holder role). Two hard
constraints make this a one-shot:

- Apple lets an account create **at most five** Developer ID Application
  certificates, ever. Revoking one does **not** free a slot.
- Revoking a certificate invalidates Gatekeeper for **every installed build**
  signed with it, not just future ones.

So create exactly **one**, then immediately open **Keychain Access** and export
the private key as a `.p12` to safe, backed-up storage. The certificate is
valid five years; the `.p12` is the only way to reconstitute the identity on
another machine without burning a second slot.

### 2. Developer portal registrations

- **App Group:** `group.app.kernova`.
- **App IDs**, each with the App Groups capability configured to include that
  group: `app.kernova`, `app.kernova.macosagent`, `app.kernova.fileprovider`,
  `app.kernova.macosagent.fileprovider`.

### 3. Two manual Developer ID provisioning profiles

Created by hand at **portal → Profiles → + → Distribution → Developer ID**
(not generated by Xcode):

| Profile name | App ID |
|---|---|
| `Kernova macOS Agent Developer ID` | `app.kernova.macosagent` |
| `Kernova macOS Agent File Provider Developer ID` | `app.kernova.macosagent.fileprovider` |

Both must: include the **App Groups** entitlement for `group.app.kernova`
(portal support for iOS-style groups in Mac Developer ID profiles landed
2025-02-21); embed the **Developer ID Application** certificate; and contain
`ProvisionsAllDevices` — i.e. they are **not** device-locked. The
non-device-locked property is what lets the guest agent validate and launch
inside an *unregistered* guest VM (a device-locked Development profile is
refused there by `fileproviderd`). They expire ~18 years out. Install them
locally via **Xcode → Settings → Accounts → Download Manual Profiles**.

**If the certificate is ever regenerated, both profiles must be regenerated**
to embed the new certificate.

## Why the agent's profiles are manual but the app's are not

The guest agent — `Kernova Guest Agent.app` plus its embedded
`KernovaMacOSAgentFileProvider.appex` — is packaged into a disk image at build
time by the **Package Guest Agent DMG** build phase, which writes
`Resources/KernovaMacOSAgent.dmg` inside `Kernova.app`. Xcode's export-time
re-signing walks nested bundle *code*, but it **cannot reach inside a DMG
resource**: whatever signature the agent carried at archive time is exactly
what ships to guests.

Therefore the agent target and its File Provider extension must be fully,
correctly signed at **build** time. Their Release configs are **Manual**, with
`PROVISIONING_PROFILE_SPECIFIER` set to the two profile names above,
`ENABLE_HARDENED_RUNTIME = YES`, and `OTHER_CODE_SIGN_FLAGS = "--timestamp"`.
Notarization requires a hardened runtime and a secure timestamp on every
binary, and since export re-signing can't add either inside the DMG, they have
to be baked in at build time.

The outer app, the host `KernovaFileProvider.appex`, `KernovaQuickLook.appex`,
and `KernovaRelaunchHelper` all keep **Automatic / Apple Development** at build
time and receive their Developer ID signatures from the Organizer export. The
app and host File Provider targets set `REGISTER_APP_GROUPS = YES` so Xcode's
automatic signing generates/updates Developer ID profiles carrying the
`group.app.kernova` authorization at export (Xcode 16.3+ behavior).

One CLI consequence of `REGISTER_APP_GROUPS`: a terminal
`xcodebuild -configuration Release build` fails with *"Provisioning profile
'Mac Team Provisioning Profile' doesn't include the App Groups capability"*
until the cached team profiles are regenerated. Pass
`-allowProvisioningUpdates` (needed once per profile change; the Xcode GUI
does this implicitly). Xcode also injects `com.apple.security.get-task-allow`
into *build*-action products for debugger attachment — the agent targets set
`CODE_SIGN_INJECT_BASE_ENTITLEMENTS = NO` in Release so the agent baked into
the DMG is notarization-clean regardless of whether it came from a `build` or
an `archive` action.

All six product targets set `ENABLE_HARDENED_RUNTIME = YES` in Release. The
agent and `KernovaRelaunchHelper` each set `SKIP_INSTALL = YES` so the archive
contains only `Kernova.app` — either one installing as a second product turns
the archive into a generic *Other Items* archive with **Distribute App
disabled**. (The three embedded appexes already carry `SKIP_INSTALL = YES` by
default; the agent and helper are standalone app/tool targets that otherwise
install.)

## Release flow

1. **Version.** Decide whether to bump `MARKETING_VERSION` for the app
   (currently `0.9.1`). The guest agent has its **own** version (currently
   `0.31.2`) — bump it only when agent behavior changed, per the guest-agent
   versioning conventions in CLAUDE.md. `CFBundleVersion` is automatic (git
   commit count) and needs no manual edit.
2. **Archive.** In Xcode, select the **Kernova** scheme, then **Product →
   Archive** (archiving builds Release).
3. **Confirm the archive type.** In the Organizer, verify the new archive
   appears as a **macOS app** archive, not *Other Items*. If it's *Other
   Items*, a `SKIP_INSTALL` regressed on the agent or `KernovaRelaunchHelper`.
4. **Distribute.** **Distribute App → Direct Distribution → Automatically
   manage signing.** Xcode re-signs the app and its nested code with Developer
   ID, submits to the notary service, staples the ticket on success, and
   enables **Export** (usually within minutes).

## Verification checklist

Run against the **exported** `Kernova.app`:

```bash
# Notarized and Gatekeeper-accepted
spctl -a -vv Kernova.app
# → "accepted", "source=Notarized Developer ID"

# Correct signing authority
codesign -dvvv Kernova.app
# → Authority=Developer ID Application: Nicholas Lonsinger (8MT4P4GZL2)

# Entitlements: app group present, debug entitlement absent
codesign -d --entitlements - Kernova.app
# → shows group.app.kernova and NO com.apple.security.get-task-allow

# Deep structural verification
codesign -dvv --verify --strict --deep Kernova.app
```

Then inspect the agent **inside the DMG** (the one bundle export re-signing
can't touch):

```bash
hdiutil attach Kernova.app/Contents/Resources/KernovaMacOSAgent.dmg
codesign -dvvv "/Volumes/Kernova Guest Agent/Kernova Guest Agent.app"
# → Developer ID Application authority
# → flags=0x10000(runtime)          (hardened runtime present)
# → "Timestamp=" line present       (secure timestamp present)
# → an embedded.provisionprofile with ProvisionsAllDevices
hdiutil detach "/Volumes/Kernova Guest Agent"
```

Finally, a **functional pass** on the disposable **"macOS TEST"** VM (never the
VM named "macOS", which may hold real data):

- Gatekeeper first-launch of the exported app succeeds.
- Inside the guest, **Install Guest Agent** from the bundled DMG comes up with
  **no** "access data from other apps" consent prompt (this is the payoff of
  the non-device-locked `ProvisionsAllDevices` profiles).
- **Copy to Mac** (the guest→host File Provider flow) materializes files on the
  host, and host↔guest clipboard sync works in both directions.

## Package for distribution

```bash
ditto -c -k --keepParent Kernova.app Kernova-<version>.zip
```

The notarization ticket is **stapled to the `.app`**, so a plain zip is a valid
distribution artifact — Gatekeeper validates offline from the stapled ticket.
